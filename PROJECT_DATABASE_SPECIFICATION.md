## Обзор проекта

Система управления персоналом и развитием компетенций, включающая:
- Управление пользователями и ролями
- Система оценки 360 градусов
- Опросы профессиональных навыков
- Карьерные треки и развитие
- Встречи 1:1
- Диагностические этапы
- Планы развития и задачи

## Функции базы данных

### 1. Функции проверки ролей и прав

#### `has_role(_user_id uuid, _role app_role) RETURNS boolean`
Проверяет, имеет ли пользователь определенную роль.

**Параметры:**
- `_user_id` - ID пользователя
- `_role` - Роль для проверки

**Возвращает:** `true` если пользователь имеет указанную роль.

#### `has_permission(_user_id uuid, _permission_name text) RETURNS boolean`
Проверяет, имеет ли пользователь определенное разрешение.

**Параметры:**
- `_user_id` - ID пользователя
- `_permission_name` - Название разрешения

**Возвращает:** `true` если пользователь имеет указанное разрешение.

#### `get_user_role(_user_id uuid) RETURNS app_role`
Получает роль пользователя.

**Параметры:**
- `_user_id` - ID пользователя

**Возвращает:** Роль пользователя.

#### `is_manager_of(_manager_id uuid, _employee_id uuid) RETURNS boolean`

**Возвращает:** `true` если текущий пользователь - администратор.

**Параметры:**
- `target_user_id` - ID целевого пользователя

### 8. Функции работы с диагностическими этапами

#### `calculate_diagnostic_stage_progress(stage_id_param uuid) RETURNS numeric`
Вычисляет процент завершения диагностического этапа.

**Параметры:**
- `stage_id_param` - ID этапа

**Логика:**
- Подсчитывает общее количество участников
- Каждый участник должен завершить 2 опроса (навыки + 360)
- Подсчитывает завершенные опросы
- Вычисляет процент: (завершено / требуется) * 100

**Возвращает:** Процент завершения (0-100).

#### `update_diagnostic_stage_status() RETURNS trigger`
Обновляет статус и прогресс диагностического этапа.

**Логика:**
- При добавлении результатов оценки находит активный этап для пользователя
- Пересчитывает прогресс этапа
- Обновляет статус: 'setup' (0%), 'assessment' (1-99%), 'completed' (100%)

**Используется в:** Триггеры на `survey_360_results` и `skill_survey_results`.

#### `update_diagnostic_stage_on_participant_add() RETURNS trigger`
Обновляет прогресс этапа при добавлении участника.

**Логика:**
- При добавлении нового участника в этап
- Пересчитывает прогресс этапа
- Обновляет статус этапа

**Используется в:** Триггер на `diagnostic_stage_participants`.

### 9. Функции аудита

#### `log_admin_action(...) RETURNS uuid`
Логирует действия администратора.

**Параметры:**
- `_admin_id` - ID администратора
- `_target_user_id` - ID целевого пользователя
- `_action_type` - Тип действия
- `_field` - Измененное поле (опционально)
- `_old_value` - Старое значение (опционально)
- `_new_value` - Новое значение (опционально)
- `_details` - Дополнительные детали в JSON (опционально)

**Возвращает:** ID созданной записи лога.

## Бизнес-логика и процессы

### 1. Процесс оценки 360 градусов

**Этапы:**

1. **Создание назначений**
   - HR или менеджер создает назначения оценки 360
   - Указывает кто кого оценивает (`survey_360_assignments`)
   - Автоматически создается задача для оценивающего (триггер `create_task_for_assignment`)

2. **Прохождение оценки**
   - Оценивающий заполняет опрос
   - Результаты сохраняются в `survey_360_results`
   - Автоматически устанавливается период оценки (триггер `set_evaluation_period`)

3. **Обновление данных**
   - При сохранении результатов обновляются `user_qualities` (триггер `update_user_qualities_from_survey`)
   - Обновляется статус назначения на 'выполнено' (триггер `update_assignment_on_survey_completion`)
   - Задача помечается как выполненная
   - Создаются агрегированные результаты в `user_assessment_results` (триггер `insert_assessment_results`)

4. **Просмотр результатов**
   - Пользователь может видеть свои результаты
   - Менеджер может видеть результаты подчиненных
   - HR и администраторы видят все результаты

### 2. Процесс опроса профессиональных навыков

**Этапы:**

1. **Создание назначений**
   - HR или менеджер создает назначения опроса
   - Указывает кто кого оценивает (`skill_survey_assignments`)
   - Автоматически создается задача (триггер `create_task_for_skill_assignment`)

2. **Прохождение опроса**
   - Оценивающий заполняет опрос
   - Результаты сохраняются в `skill_survey_results`
   - Автоматически устанавливается период оценки

3. **Обновление данных**
   - Обновляются `user_skills` (триггер `update_user_skills_from_survey`)
   - Обновляется статус назначения (триггер `update_skill_assignment_on_survey_completion`)
   - Задача помечается как выполненная
   - Создаются агрегированные результаты

### 3. Процесс диагностических этапов

**Этапы:**

1. **Создание этапа**
   - HR создает диагностический этап (`diagnostic_stages`)
   - Указывает период, даты начала, окончания и дедлайн
   - Статус этапа: 'setup'

2. **Добавление участников**
   - HR добавляет участников этапа (`diagnostic_stage_participants`)
   - Для каждого участника автоматически создается задача (триггер `create_diagnostic_task_for_participant`)
   - Обновляется прогресс этапа (триггер `update_diagnostic_stage_on_participant_add`)

3. **Прохождение диагностики**
   - Участники проходят опрос навыков и оценку 360
   - При каждом завершении опроса обновляется прогресс этапа (триггер `update_diagnostic_stage_status`)
   - Статус меняется: 'setup' → 'assessment' → 'completed'

4. **Завершение этапа**
   - Когда все участники завершили оба опроса, прогресс = 100%
   - Статус этапа: 'completed'

### 4. Процесс встреч 1:1

**Этапы:**

1. **Создание этапа**
   - HR создает этап встреч (`meeting_stages`)
   - Указывает период, даты и дедлайн
   - Этап активен (`is_active = true`)

2. **Добавление участников**
   - HR добавляет участников этапа (`meeting_stage_participants`)
   - Для каждого участника создается задача (триггер `create_meeting_task_for_participant`)

3. **Заполнение формы встречи**
   - Сотрудник создает встречу (`one_on_one_meetings`)
   - Заполняет поля: цели, энергия, решения предыдущих встреч, проблемы
   - Статус: 'draft'

4. **Отправка на проверку**
   - Сотрудник отправляет форму менеджеру
   - Статус: 'submitted'
   - Устанавливается `submitted_at`

5. **Проверка менеджером**
   - Менеджер просматривает форму
   - Может утвердить ('approved') или вернуть ('returned')
   
   **При утверждении:**
   - Статус: 'approved'
   - Устанавливается `approved_at`
   - Задача помечается как выполненная (триггер `update_meeting_task_status`)
   
   **При возврате:**
   - Статус: 'returned'
   - Указывается причина возврата
   - Устанавливается `returned_at`
   - Сотрудник может исправить и отправить снова

6. **Добавление решений**
   - Во время встречи создаются решения (`meeting_decisions`)
   - Решения можно помечать как выполненные
   - Решения используются на следующей встрече

### 5. Процесс карьерного развития

**Логика карьерных треков:**

1. **Создание трека**
   - Администратор создает карьерный трек (`career_tracks`)
   - Указывает тип трека, целевую позицию, длительность
   - Добавляет шаги трека (`career_track_steps`)
   - Каждый шаг привязан к грейду и имеет порядок

2. **Назначение трека пользователю**
   - Пользователь получает трек на основе текущей позиции и грейда
   - Система показывает текущий шаг и следующие шаги

3. **Требования для перехода**
   - Каждый грейд имеет требования по навыкам (`grade_skills`)
   - Каждый грейд имеет требования по качествам (`grade_qualities`)
   - Пользователь видит свой прогресс по каждому требованию

4. **Планы развития**
   - На основе пробелов создается план развития (`development_plans`)
   - План содержит задачи из библиотеки (`development_tasks`)
   - Задачи привязаны к конкретным навыкам/качествам и уровням

### 6. Права доступа (RLS Policies)

**Общие принципы:**

1. **Администраторы (admin)**
   - Полный доступ ко всем данным
   - Могут изменять справочники
   - Могут управлять пользователями и ролями

2. **HR специалисты**
   - Могут создавать этапы (встреч, диагностики)
   - Могут добавлять участников
   - Могут просматривать результаты всех сотрудников
   - Могут создавать планы развития

3. **Менеджеры**
   - Могут просматривать данные своих подчиненных
   - Могут утверждать/возвращать встречи 1:1
   - Могут назначать оценки для подчиненных

4. **Сотрудники**
   - Могут просматривать свои данные
   - Могут проходить назначенные опросы
   - Могут создавать и заполнять встречи 1:1
   - Могут просматривать свой карьерный трек и планы развития

**Специфичные политики:**

- `survey_360_results`: Пользователь видит свои оценки, оценивающий видит то, что оценивал, менеджер видит оценки подчиненных
- `skill_survey_results`: Аналогично оценке 360
- `one_on_one_meetings`: Доступ только участникам встречи (сотрудник + менеджер)
- `tasks`: Пользователь видит только свои задачи
- `user_skills`, `user_qualities`: Пользователь и его менеджеры могут просматривать

### 7. Периоды оценки

**Логика периодов:**

- Год делится на два полугодия: H1 (январь-июнь) и H2 (июль-декабрь)
- Автоматически определяется при создании результата оценки
- Формат: 'H1_2025', 'H2_2025'
- Используется для группировки и сравнения результатов по периодам
- Позволяет отслеживать динамику развития

## Ключевые особенности системы

1. **Автоматизация**
   - Автоматическое создание задач при назначении опросов
   - Автоматическое обновление профилей пользователей при прохождении оценок
   - Автоматический расчет периодов оценки
   - Автоматическое обновление прогресса диагностических этапов

2. **Гибкость**
   - Настраиваемые грейды и требования
   - Гибкие карьерные треки
   - Кастомизируемые вопросы и варианты ответов

3. **Отслеживание**
   - Полный аудит действий администраторов
   - Логирование активности
   - Отслеживание прогресса по задачам и этапам

4. **Безопасность**
   - Row-Level Security на всех таблицах
   - Разграничение доступа по ролям
   - Анонимные комментарии в оценке 360

5. **Аналитика**
   - Агрегированные результаты оценок
   - Сравнение результатов по периодам
   - Gap-анализ (сравнение текущего и целевого уровня)
   - Отчеты по прогрессу развития
