## 1. Область применения и границы системы

### 1.1 Назначение

**Milu** — корпоративная HRM-платформа для управления компетенциями сотрудников, включающая:
- Диагностику 360° (оценка soft/hard skills коллегами, руководителем, самооценка)
- Встречи 1:1 (регулярные встречи сотрудник-руководитель)
- Планы развития (ИПР)
- Управление карьерными треками и грейдами

### 1.2 Технологический стек

| Слой | Технология |
|------|------------|
| Frontend | React 18, TypeScript, Vite |
| Стили | Tailwind CSS, shadcn/ui |
| Состояние | TanStack React Query |
| Роутинг | React Router DOM 6.x |
| Backend | Supabase (PostgreSQL 15+, Edge Functions, RLS) |
| Авторизация | Supabase Auth (email/password) |
| Фоновые задачи | pg_cron (PostgreSQL extension) |

### 1.3 Границы системы

**Включено:**
- Управление пользователями и ролями
- Диагностические этапы с участниками
- Опросы 360 (soft skills) и оценка навыков (hard skills)
- Система задач (tasks)
- Встречи 1:1 с решениями
- Справочники (грейды, навыки, качества, должности)
- Импорт данных из Excel

**НЕ включено:**
- Интеграция с внешними HR-системами (1С, SAP)
- Мобильные приложения (только веб)
- Real-time уведомления (push, email)

## 3. Авторизация и сессии

### 3.1 Как работает сейчас

- **Механизм**: Supabase Auth с email/password
- **Хранение сессии**: В памяти (Supabase JS SDK), автоматический refresh token
- **Cookie Consent**: Обязательное согласие при первом входе (поля `cookies_consent`, `cookies_consent_at` в `users`)

### 3.2 Где в коде

| Компонент | Путь | Назначение |
|-----------|------|------------|
| AuthContext | `src/contexts/AuthContext.tsx` | Глобальный контекст пользователя |
| AuthGuard | `src/components/AuthGuard.tsx` | HOC для защиты роутов |
| AuthPage | `src/pages/AuthPage.tsx` | Страница входа |

### 3.3 Сценарий авторизации

```typescript
// src/pages/AuthPage.tsx:69-108
const handleLogin = async (e: React.FormEvent) => {
  const { data, error } = await supabase.auth.signInWithPassword({
    email,
    password,
  });
  if (data.session) {
    // Обновляем cookie consent если нужно
    if (needsCookieConsent && cookiesConsent) {
      await supabase.from('users').update({
        cookies_consent: true,
        cookies_consent_at: new Date().toISOString()
      }).eq('id', data.session.user.id);
    }
    navigate('/');
  }
};
```

### 3.4 Обработка истечения сессии

- **Supabase SDK** автоматически обновляет access token через refresh token
- **При ошибке refresh**: `onAuthStateChange` с event `SIGNED_OUT` → редирект на `/auth`
- **AuthGuard**: проверяет `loading` и `isAuthenticated`, показывает спиннер или редиректит

### 3.5 Критические инварианты

1. Все роуты кроме `/auth` обёрнуты в `<AuthGuard>`
2. `AuthContext` загружает роль асинхронно после получения session
3. Cookie consent проверяется по email ДО логина (onBlur)

### 3.6 Как проверить

```sql
-- Проверить согласие на cookies
SELECT id, email, cookies_consent, cookies_consent_at 
FROM users 
WHERE email = 'test@example.com';
```

| RLS policies (БД) | Основной enforcement |

### 4.6 SQL-функция проверки прав

```sql
-- has_permission(_permission_name text) RETURNS boolean
-- SECURITY DEFINER: обходит RLS для внутренней проверки
CREATE OR REPLACE FUNCTION public.has_permission(_permission_name text)
RETURNS boolean AS $$
DECLARE
  _user_id uuid;
  _has_perm boolean;
BEGIN
  _user_id := auth.uid();
  IF _user_id IS NULL THEN RETURN false; END IF;
  
  SELECT EXISTS (
    SELECT 1 FROM role_permissions rp
    JOIN user_roles ur ON ur.role = rp.role
    JOIN permissions p ON p.id = rp.permission_id
    WHERE ur.user_id = _user_id AND p.name = _permission_name
  ) INTO _has_perm;
  
  RETURN _has_perm;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

### 4.7 Критические инварианты

1. **Источник истины** — RLS policies в БД, не фронтенд
2. Проверки во фронтенде — только для UX (скрытие кнопок)
3. Edge Functions проверяют права ДО выполнения операций

```

│ (none)  │────────────>│  setup  │───────────────────>│ active  │

                                                            ▼

                                                    │   false)      │
                                                    └───────────────┘
```

### 7.3 Автозакрытие этапов

**Механизм**: pg_cron job каждые 5 минут

```sql
-- Запуск job
SELECT cron.schedule(
  'finalize-expired-stages',
  '*/5 * * * *',
  $$SELECT public.check_and_finalize_expired_stages()$$
);
```

**Функция check_and_finalize_expired_stages():**
1. Находит все `parent_stages` где `end_date < NOW()` AND `is_active = true`
2. Для каждого вызывает `finalize_expired_stage(p_stage_id)`

**Функция finalize_expired_stage(p_stage_id):**

```sql
-- Псевдокод
BEGIN
  -- 1. Деактивация parent_stage
  UPDATE parent_stages SET is_active = false WHERE id = p_stage_id;
  
  -- 2. Деактивация diagnostic_stages
  UPDATE diagnostic_stages SET is_active = false WHERE parent_id = p_stage_id;
  
  -- 3. Обновление meeting_stages
  UPDATE meeting_stages SET updated_at = NOW() WHERE parent_id = p_stage_id;
  
  -- 4. Snapshot и expire assignments
  UPDATE survey_360_assignments
  SET status = 'expired'
  WHERE diagnostic_stage_id IN (SELECT id FROM diagnostic_stages WHERE parent_id = p_stage_id)
    AND status NOT IN ('completed', 'approved', 'rejected');
  
  -- 5. Expire tasks
  UPDATE tasks
  SET status = 'expired'
  WHERE diagnostic_stage_id IN (SELECT id FROM diagnostic_stages WHERE parent_id = p_stage_id)
    AND status NOT IN ('completed');
  
  -- 6. Snapshot meetings
  UPDATE one_on_one_meetings
  SET status_at_stage_end = status,
      stage_end_snapshot_at = NOW(),
      status = 'expired'
  WHERE stage_id IN (SELECT id FROM meeting_stages WHERE parent_id = p_stage_id)
    AND status NOT IN ('approved');
END;
```

## 8. Диагностика 360 (end-to-end)

### 8.1 Полный флоу

```
1. СОЗДАНИЕ ЭТАПА
   Admin/HR → StagesPage → создаёт parent_stage + diagnostic_stage
   
2. ДОБАВЛЕНИЕ УЧАСТНИКОВ
   Admin/HR → addParticipants() → INSERT diagnostic_stage_participants
   → TRIGGER handle_diagnostic_participant_added
     → создаёт survey_360_assignments (self, manager)
     → создаёт tasks (peer_selection, self_assessment, manager_assessment)

3. ВЫБОР РЕСПОНДЕНТОВ (peer_selection)
   Employee → TasksManager → PeerSelectionButton → ColleagueSelectionDialog
   → создаёт survey_360_assignments (peer, status=pending)
   → завершает task (peer_selection → completed)
   → вызывает Edge Function create-peer-approval-task
     → создаёт task (peer_approval) для manager

4. УТВЕРЖДЕНИЕ РЕСПОНДЕНТОВ (peer_approval)
   Manager → ManagerRespondentApproval → approve/reject assignments
   → UPDATE survey_360_assignments SET status = 'approved' | 'rejected'
   → вызывает Edge Function create-peer-evaluation-tasks
     → создаёт tasks для каждого approved peer

5. ЗАПОЛНЕНИЕ ОПРОСОВ
   Any evaluator → UnifiedAssessmentPage
   → автосохранение с is_draft=true
   → финальное сохранение: is_draft=false
   → UPDATE tasks SET status = 'completed'
   → UPDATE survey_360_assignments SET status = 'completed'

6. ПРОСМОТР РЕЗУЛЬТАТОВ
   Employee/Manager/HR → AssessmentResultsPage
   → useCorrectAssessmentResults(userId, filterType, stageId)
   → агрегация по soft_skill_results + hard_skill_results WHERE is_draft=false
```

### 8.2 Типы назначений (assignment_type)

| Тип | Кто оценивает | Кого оценивает |
|-----|---------------|----------------|
| `self` | Сотрудник | Себя |
| `manager` | Руководитель | Подчинённого |
| `peer` | Коллега | Сотрудника |

## 9. Встречи 1:1 (end-to-end)

### 9.1 Полный флоу

```
1. СОЗДАНИЕ ЭТАПА ВСТРЕЧ
   Admin/HR → создаёт meeting_stage привязанный к parent_stage

2. ДОБАВЛЕНИЕ УЧАСТНИКОВ
   Admin/HR → addParticipants() → INSERT meeting_stage_participants
   → TRIGGER создаёт one_on_one_meetings со статусом 'draft'

3. ЗАПОЛНЕНИЕ ФОРМЫ (сотрудник)
   Employee → MeetingsPage → MeetingForm
   → поля: goal_and_agenda, energy_gained, energy_lost, stoppers
   → сохранение: UPDATE one_on_one_meetings

4. ОТПРАВКА НА УТВЕРЖДЕНИЕ
   Employee → submitMeeting()
   → UPDATE SET status = 'submitted', submitted_at = NOW()

5. РАССМОТРЕНИЕ РУКОВОДИТЕЛЕМ
   Manager → MeetingsPage
   → Варианты:
     a) approve → status = 'approved', approved_at = NOW()
     b) return → status = 'returned', return_reason = '...'

6. ДОРАБОТКА (при return)
   Employee → исправляет → снова submit

7. ДОБАВЛЕНИЕ РЕШЕНИЙ
   Both → MeetingDecisions
   → INSERT meeting_decisions (decision_text, is_completed)
```

### 9.2 Статусы встречи

| Статус | Описание | Кто может изменить |
|--------|----------|-------------------|
| `draft` | Черновик, заполняется сотрудником | Employee |
| `submitted` | Отправлено на утверждение | Employee → Manager |
| `returned` | Возвращено на доработку | Manager → Employee |
| `approved` | Утверждено | Manager (финал) |
| `expired` | Закрыто автоматически | pg_cron |

## 12. Расчёты и агрегации

### 12.1 Правила расчёта AVG

**Включается:**
- `is_draft = false`
- `is_skip = false` ИЛИ `is_skip IS NULL`
- Привязка к конкретному `diagnostic_stage_id`

**Исключается:**
- Черновики (`is_draft = true`)
- Пропущенные вопросы (`is_skip = true`)

```typescript
const aggregateScores = (scores: RawScore[], userId: string) => {
  const selfScores = scores.filter(s => s.evaluating_user_id === userId);
```

## 19. SQL-диагностики (read-only)

### 19.1 Активные этапы

```sql
SELECT ps.id, ps.period, ps.start_date, ps.end_date, ps.is_active,
       ds.id as diagnostic_stage_id, ds.is_active as ds_active,
       ms.id as meeting_stage_id
FROM parent_stages ps
LEFT JOIN diagnostic_stages ds ON ds.parent_id = ps.id
LEFT JOIN meeting_stages ms ON ms.parent_id = ps.id
WHERE ps.is_active = true
ORDER BY ps.start_date DESC;
```

```sql
SELECT ds.id as stage_id,
       a.status,
       a.assignment_type,
       COUNT(*) as count

GROUP BY ds.id, a.status, a.assignment_type
ORDER BY ds.id, a.status;
```
