## 1. Общее описание проекта

### Назначение системы

Система управления компетенциями и развитием сотрудников предназначена для:
- Оценки профессиональных навыков (hard skills)
- Оценки личностных качеств (soft skills) методом 360°
- Планирования карьерного развития
- Проведения встреч 1:1 между сотрудниками и руководителями
- Формирования индивидуальных планов развития
- Управления задачами развития
- Аналитики компетенций на уровне сотрудников, команд и организации

### Ключевые модули

#### 1. Диагностика компетенций
- **Unified Assessment** - единая страница для прохождения оценки (навыки + качества)
- Самооценка, оценка руководителем, оценка коллегами (360°)
- Агрегация результатов в `user_assessment_results`
- Поддержка черновиков (`is_draft`)
- Периоды оценки (H1/H2)

#### 2. Встречи 1:1
- Заполнение форм сотрудником и руководителем
- Утверждение руководителем
- Возврат на доработку
- Принятие решений по итогам встреч
- Создание задач на основе решений

#### 3. Карьерное развитие
- Карьерные треки с шагами (грейдами)
- Расчет GAP-анализа (разрыв между текущим и целевым уровнем)
- Рекомендация карьерных треков на основе текущих компетенций
- Отслеживание прогресса по треку

#### 4. Планы развития и задачи
- Автоматическое создание задач на основе GAP-анализа
- Edge Function `generate-development-tasks` для создания персонализированных задач
- Типы задач: diagnostic, meeting, development, peer_evaluation
- Статусы: pending, in_progress, completed

## 3. Авторизация

### Supabase Auth

**Метод авторизации:** Email + пароль

**Кастомная система сессий:**
- Таблица `admin_sessions` хранит активные сессии
- Срок действия сессии: 24 часа
- Edge Function `custom-login` для входа

### Логика определения текущего пользователя

**AuthContext** (`src/contexts/AuthContext.tsx`):
```typescript
interface AuthUser {
  id: string;
  full_name: string;
  email: string;
  role: 'admin' | 'hr_bp' | 'manager' | 'employee';
  permissions?: string[];
}
```

**Процесс:**
1. При загрузке приложения проверяется наличие активной сессии в `admin_sessions`
2. Если сессия валидна, загружаются данные пользователя из `users`
3. Загружается роль из `user_roles`
4. Загружаются права доступа из `user_effective_permissions`
5. Формируется объект `AuthUser` и сохраняется в контексте

**Функция получения текущего пользователя (SQL):**
```sql
CREATE FUNCTION get_current_session_user() RETURNS uuid
SECURITY DEFINER
AS $$
  SELECT user_id 
  FROM admin_sessions 
  WHERE expires_at > now() 
  ORDER BY created_at DESC 
  LIMIT 1;
$$ LANGUAGE sql STABLE;
```

### Роли и доступы на фронте

**Защита маршрутов:** `AuthGuard` компонент проверяет `isAuthenticated` перед рендерингом

**Проверка прав:** Хук `usePermission` проверяет наличие конкретного права у пользователя

### Токены

**JWT токены** используются через Supabase Auth:
- Access token (короткий срок действия)
- Refresh token (автоматическое обновление)

**Хранение:**
- localStorage (через Supabase клиент)
- Автоматическая синхронизация между вкладками

### RLS-политики для каждой таблицы

**Общий принцип:**
- Все таблицы защищены Row Level Security (RLS)
- Политики используют функции `has_permission()`, `is_owner()`, `is_users_manager()`
- Администраторы имеют полный доступ через `has_permission('admin.manage_all')`

**Ключевые политики:**

**users:**
- SELECT: Все аутентифицированные пользователи (базовая информация)
- INSERT: Только admin/hr_bp
- UPDATE: Только admin/hr_bp
- DELETE: Только admin

**survey_360_assignments:**
- SELECT: Оцениваемый, оценивающий, их руководители, HR BP
- INSERT: HR BP, руководитель (для подчиненных)
- UPDATE: HR BP, руководитель (статусы)
- DELETE: HR BP, admin

**hard_skill_results / soft_skill_results:**
- SELECT: Владелец результата, руководитель, HR BP
- INSERT: Только оценивающий пользователь
- UPDATE: Только оценивающий (если is_draft = true)
- DELETE: Admin

**tasks:**
- SELECT: Владелец задачи, его руководитель, HR BP
- INSERT: HR BP, руководитель, система (через функции)
- UPDATE: Владелец (статус), HR BP, руководитель
- DELETE: Admin

**one_on_one_meetings:**
- SELECT: Сотрудник, руководитель, HR BP
- INSERT: Система (через триггеры)
- UPDATE: Сотрудник (черновик), руководитель (утверждение)
- DELETE: Admin

**user_assessment_results:**
- SELECT: Владелец результатов, руководитель, HR BP
- INSERT/UPDATE/DELETE: Только через триггеры (нет прямого доступа)

### Фоновые процессы и автоматика

#### Создание задач диагностики
**Когда:** При создании участника диагностики (`diagnostic_stage_participants`)  
**Что происходит:**
1. Создается задача типа `diagnostic` для участника
2. Проверяется наличие руководителя
3. Если руководитель есть, создается assignment типа `manager`
4. Автоматически создается assignment типа `self` для самооценки

#### Агрегация результатов
**Когда:** После сохранения результатов (`hard_skill_results`, `soft_skill_results`)  
**Функция:** `aggregate_hard_skill_results()`, `aggregate_soft_skill_results()`  
**Логика:**
```sql
-- Для каждого навыка/качества пользователя:
1. Берем все оценки (не черновики) за период
2. Фильтруем по assignment_type:
   - self → self_assessment
   - manager → manager_assessment
   - peer → peers_average (AVG)
3. Подсчитываем total_responses
4. UPSERT в user_assessment_results
```

#### Генерация задач развития
**Edge Function:** `generate-development-tasks`  
**Вызов:** Вручную через UI или API  
**Логика:**
1. Получение GAP-анализа через `calculate_career_gap()`
2. Получение рекомендаций через `get_recommended_development_tasks()`
3. Создание задач типа `development` на основе рекомендаций
4. Приоритизация задач по размеру GAP

## 7. Бизнес-логика

### Диагностика (полный процесс)

#### Шаг 1: Создание этапа диагностики
**Роль:** Admin/HR BP  
**Страница:** `/admin/diagnostics`  
**Действие:**
1. Создается `parent_stage` с периодом и датами
2. Создается `diagnostic_stage` со ссылкой на `parent_stage`
3. Добавляются участники в `diagnostic_stage_participants`

#### Шаг 2: Создание assignments и задач
**Триггер:** Автоматически при добавлении участника  
**Логика:**
1. Создается задача `diagnostic` для участника
2. Создается `survey_360_assignment` типа `self` (самооценка)
3. Если есть руководитель:
   - Создается `survey_360_assignment` типа `manager`
   - Создается задача для руководителя

#### Шаг 3: Выбор коллег для оценки 360°
**Роль:** Employee  
**Страница:** `/unified-assessment/:assignmentId`  
**Компонент:** `ColleagueSelectionDialog`  
**Действие:**
1. Сотрудник выбирает коллег (обычно 3-5)
2. Создаются `survey_360_assignments` типа `peer` со статусом `pending`
3. Ожидается утверждение руководителем

#### Шаг 4: Утверждение списка коллег
**Роль:** Manager  
**Страница:** `/team`  
**Компонент:** `ManagerRespondentApproval`  
**Действие:**
1. Руководитель видит список коллег на утверждение
2. Может утвердить (`approved`) или отклонить (`rejected`)
3. При утверждении создаются задачи типа `peer_evaluation` для коллег
4. Статус assignment меняется на `approved`

#### Шаг 5: Прохождение оценки (Unified Assessment)
**Роль:** Employee/Manager/Peer  
**Страница:** `/unified-assessment/:assignmentId`  
**Компонент:** `UnifiedAssessmentPage`  
**Логика:**
1. Загружаются вопросы для грейда оцениваемого:
   - `hard_skill_questions` (по навыкам из `grade_skills`)
   - `soft_skill_questions` (по качествам из `grade_qualities`)
2. Пользователь отвечает на вопросы
3. Может добавлять комментарии
4. Опция анонимности комментариев (только для peer)
5. Сохранение как черновик (`is_draft = true`)
6. Финальная отправка (`is_draft = false`)

#### Шаг 6: Агрегация результатов
**Триггер:** Автоматически при сохранении не-черновика  
**Функции:**
- `aggregate_hard_skill_results()`
- `aggregate_soft_skill_results()`

**Результат:** Заполняется `user_assessment_results` с:
- `self_assessment` - самооценка
- `manager_assessment` - оценка руководителя
- `peers_average` - средняя оценка коллег

#### Шаг 7: Просмотр результатов
**Роль:** Employee (свои), Manager (команды), HR BP (все)  
**Страницы:**
- `/assessment/results/:userId` - детальные результаты
- `/profile` - профиль с виджетами
- `/manager-reports` - отчеты для руководителей

### Unified Assessment (детально)

**Компонент:** `UnifiedAssessmentPage`  
**Маршрут:** `/unified-assessment/:assignmentId`

**Функциональность:**
1. **Проверка черновика:** Хук `useAssignmentDraftStatus` проверяет наличие черновика
2. **Загрузка данных:**
   - Assignment из `survey_360_assignments`
   - Оцениваемый пользователь и его грейд
   - Навыки грейда из `grade_skills`
   - Качества грейда из `grade_qualities`
   - Вопросы: `hard_skill_questions` (по `skill_id IN skillIds`)
   - Вопросы: `soft_skill_questions` (по `quality_id IN qualityIds`)
   - Варианты ответов: `hard_skill_answer_options`, `soft_skill_answer_options`
   - Существующие ответы (если черновик)
3. **Сохранение:**
   - **Черновик:** `is_draft = true`, можно редактировать
   - **Финальная отправка:** `is_draft = false`, запускаются триггеры агрегации
4. **Обновление статусов:**
   - При завершении: `survey_360_assignments.status = 'completed'`
   - Триггер обновляет связанную задачу: `tasks.status = 'completed'`

### Встречи 1:1

#### Флоу встречи 1:1:

**1. Создание этапа встреч**
- Admin/HR BP создает `meeting_stage` с периодом
- Добавляет участников в `meeting_stage_participants`

**2. Автоматическое создание встреч**
- Триггер создает записи `one_on_one_meetings` для каждой пары сотрудник-руководитель
- Статус: `draft`
- Создается задача для сотрудника

**3. Заполнение сотрудником**
- Страница: `/meetings`
- Сотрудник заполняет:
  - Цель и повестка
  - Что дает/забирает энергию
  - Стопперы
  - Разбор предыдущих решений
- Сохранение в статусе `draft`
- Отправка руководителю: `status = 'submitted'`, `submitted_at = NOW()`

**4. Рассмотрение руководителем**
- Руководитель видит встречу в статусе `submitted`
- Может:
  - **Утвердить:** `status = 'approved'`, `approved_at = NOW()`
  - **Вернуть:** `status = 'returned'`, заполняет `return_reason`
- Добавляет свой комментарий в `manager_comment`

**5. Создание решений**
- Компонент: `MeetingForm`
- Сотрудник или руководитель добавляет решения в `meeting_decisions`
- Решения могут порождать задачи (вручную или автоматически)

### Карьерное развитие и рекомендации

#### 1. Рекомендация карьерных треков
**Функция:** `recommend_career_tracks(p_user_id, p_limit)`

**Алгоритм:**
```sql
1. Получить все карьерные треки
2. Для каждого трека:
   a. Пройтись по всем шагам (грейдам)
   b. Для каждого грейда рассчитать GAP через calculate_career_gap()
   c. Суммировать GAP по треку (total_gap)
3. Рассчитать compatibility_score = 100 - (total_gap * 10)
4. Сортировать по compatibility_score DESC
5. Вернуть top N треков
```

**Использование:**
- Страница `/development/career-track`
- Компонент `CareerTracksWidget`

#### 2. GAP-анализ
**Функция:** `calculate_career_gap(p_user_id, p_grade_id)`

**Алгоритм:**
```sql
1. Получить требования грейда из grade_skills и grade_qualities
2. Получить текущие уровни из user_assessment_results (последний период)
3. Для каждой компетенции:
   gap = target_level - COALESCE(current_level, 0)
   is_ready = (gap <= 0)
4. Вернуть список компетенций с GAP
```

**Результат:**
```typescript
{
  competency_id: string,
  competency_name: string,
  competency_type: 'skill' | 'quality',
  current_level: number,
  target_level: number,
  gap: number,
  is_ready: boolean
}[]
```

#### 3. Генерация задач развития
**Edge Function:** `generate-development-tasks`

**Алгоритм:**
```typescript
1. Получить GAP-анализ для целевого грейда
2. Отфильтровать только компетенции с gap > 0
3. Получить рекомендованные задачи через get_recommended_development_tasks()
4. Для каждой компетенции с GAP:
   a. Найти подходящую задачу из development_tasks
   b. Создать запись в tasks с типом 'development'
   c. Установить priority на основе размера GAP:
      - gap >= 2: 'high'
      - gap >= 1: 'medium'
      - gap < 1: 'low'
5. Вернуть созданные задачи
```

**Вызов:**
- Вручную через UI (кнопка "Сгенерировать задачи")
- Автоматически при выборе карьерного трека

## 9. Ограничения и особенности

### Важные правила

#### 1. Статусы (централизованная система)
**Файл:** `src/lib/statusMapper.ts`

**БД (английские значения):**
- `pending` - Ожидает
- `approved` - Согласовано
- `completed` - Выполнено
- `in_progress` - В процессе
- `rejected` - Отклонено
- `draft` - Черновик

**UI (русские):**
- Маппинг через `getStatusLabel(dbStatus)`
- Никогда не использовать кириллические статусы в БД

**Критические ошибки (исправлены):**
- ❌ `'pending_approval'` → ✅ `'pending'`
- ❌ `'выполнено'` → ✅ `'completed'` (БД) / `'Выполнено'` (UI)

#### 2. Assignment Types
**Enum значения (БД):**
- `self` - Самооценка
- `manager` - Оценка руководителем
- `peer` - Оценка коллегой

**Инварианты:**
- Для каждого `evaluated_user_id` + `diagnostic_stage_id` должен быть ровно 1 assignment типа `self`
- Для каждого `evaluated_user_id` может быть 0-1 assignment типа `manager`
- Может быть N assignments типа `peer`

#### 3. Черновики (is_draft)
**Правила:**
- `is_draft = true` - результаты не учитываются в агрегации
- `is_draft = false` - финальные результаты, запускаются триггеры
- Можно редактировать только черновики
- После отправки (`is_draft = false`) редактирование запрещено

#### 4. Evaluation Period (автоматика)
**Формат:** `H1_YYYY` или `H2_YYYY`

**Логика:**
```sql
CREATE FUNCTION get_evaluation_period(created_date DATE) RETURNS TEXT AS $$
  SELECT CASE 
    WHEN EXTRACT(MONTH FROM created_date) <= 6 THEN 'H1_' || EXTRACT(YEAR FROM created_date)::TEXT
    ELSE 'H2_' || EXTRACT(YEAR FROM created_date)::TEXT
  END;
$$ LANGUAGE sql IMMUTABLE;
```

**Применение:**
- Триггер автоматически устанавливает при вставке в `hard_skill_results`, `soft_skill_results`
- Используется для фильтрации результатов по периодам

#### 5. Шифрование данных
**Поля (зашифрованы через Yandex Cloud Functions):**
- `users.first_name`, `last_name`, `middle_name`, `email`

**Особенности:**
- Дешифровка на клиенте через `src/lib/userDataDecryption.ts`
- Plaintext email (содержит `@`) не дешифруется
- Fallback на зашифрованные данные при ошибке API

#### 6. RLS Security
**Критично:**
- Все таблицы должны иметь RLS enabled
- Функции для проверки прав: `SECURITY DEFINER` для избежания рекурсии
- Никогда не использовать `true/true` политики в production

**Функции без рекурсии:**
```sql
CREATE FUNCTION has_role(_user_id uuid, _role app_role)
RETURNS boolean
SECURITY DEFINER
SET search_path = public
AS $$
  SELECT EXISTS (
    SELECT 1 FROM user_roles 
    WHERE user_id = _user_id AND role = _role
  )
$$ LANGUAGE sql STABLE;
```

### Бизнес-ограничения

#### 1. Диагностика
- Один активный этап диагностики на период
- Участник может быть добавлен только в активный этап
- Нельзя удалить этап с результатами
- Нельзя изменить грейд пользователя во время активной диагностики

#### 2. Встречи 1:1
- Сотрудник может создать только 1 встречу на этап
- Нельзя утвердить встречу без заполненной формы
- Возврат на доработку возможен только до утверждения

#### 3. Карьерные треки
- Пользователь может иметь только 1 активный трек
- Нельзя удалить трек с активным прогрессом
- Шаги трека должны быть упорядочены

#### 4. Задачи
- Автоматически созданные задачи (diagnostic, peer_evaluation) нельзя удалить вручную
- Задача development связана с `development_plan_tasks`
