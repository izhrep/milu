## 1. Область применения и границы системы

### 1.1 Назначение

**Milu** — корпоративная HRM-платформа для управления компетенциями сотрудников, включающая:
- Диагностику 360° (оценка soft/hard skills коллегами, руководителем, самооценка)
- Окно Джохари (AI-анализ зон восприятия по soft skills)
- Встречи 1:1 (регулярные встречи сотрудник-руководитель)
- Планы развития (ИПР)
- Управление карьерными треками и грейдами

### 1.2 Технологический стек

| Слой | Технология |
|------|------------|
| Frontend | React 18, TypeScript, Vite |
| Стили | Tailwind CSS, shadcn/ui |
| Состояние | TanStack React Query |
| Роутинг | React Router DOM 6.x |
| Backend | Supabase (PostgreSQL 15+, Edge Functions, RLS) |
| Авторизация | Supabase Auth (email/password) |
| AI | Lovable AI (Google Gemini 2.5 Flash через LOVABLE_API_KEY) |
| Фоновые задачи | pg_cron (PostgreSQL extension) |

### 1.3 Границы системы

**Включено:**
- Управление пользователями и ролями
- Диагностические этапы с участниками
- Опросы 360 (soft skills) и оценка навыков (hard skills)
- Окно Джохари с AI-генерацией отчётов
- Система задач (tasks)
- Встречи 1:1 с решениями
- Справочники (грейды, навыки, качества, должности)
- Импорт данных из Excel

**НЕ включено:**
- Интеграция с внешними HR-системами (1С, SAP)
- Мобильные приложения (только веб)
- Real-time уведомления (push, email)

## 3. Авторизация и сессии

### 3.1 Как работает сейчас

- **Механизм**: Supabase Auth с email/password
- **Хранение сессии**: В памяти (Supabase JS SDK), автоматический refresh token
- **Cookie Consent**: Обязательное согласие при первом входе (поля `cookies_consent`, `cookies_consent_at` в `users`)

### 3.2 Где в коде

| Компонент | Путь | Назначение |
|-----------|------|------------|
| AuthContext | `src/contexts/AuthContext.tsx` | Глобальный контекст пользователя |
| AuthGuard | `src/components/AuthGuard.tsx` | HOC для защиты роутов |
| AuthPage | `src/pages/AuthPage.tsx` | Страница входа |

### 3.3 Сценарий авторизации

```typescript
// src/pages/AuthPage.tsx
const handleLogin = async (e: React.FormEvent) => {
  const { data, error } = await supabase.auth.signInWithPassword({
    email,
    password,
  });
  if (data.session) {
    if (needsCookieConsent && cookiesConsent) {
      await supabase.from('users').update({
        cookies_consent: true,
        cookies_consent_at: new Date().toISOString()
      }).eq('id', data.session.user.id);
    }
    navigate('/');
  }
};
```

### 3.4 Обработка истечения сессии

- **Supabase SDK** автоматически обновляет access token через refresh token
- **При ошибке refresh**: `onAuthStateChange` с event `SIGNED_OUT` → редирект на `/auth`
- **AuthGuard**: проверяет `loading` и `isAuthenticated`, показывает спиннер или редиректит

### 3.5 Критические инварианты

1. Все роуты кроме `/auth` обёрнуты в `<AuthGuard>`
2. `AuthContext` загружает роль асинхронно после получения session
3. Cookie consent проверяется по email ДО логина (onBlur)

### 4.4 Механизм проверки прав

```
┌─────────────────────────────────────────────────────────────────┐
│ 1. RLS Policy (PostgreSQL)                                       │
│    - Основной барьер: SELECT/INSERT/UPDATE/DELETE на уровне БД  │
│    - Использует has_permission() SECURITY DEFINER функцию       │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│ 2. Edge Functions                                                │
│    - Проверка has_permission() через RPC перед операциями       │
│    - Логирование access_denied_logs при отказе                  │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│ 3. Frontend (UI)                                                 │
│    - usePermission() hook для скрытия/показа элементов          │
│    - НЕ является барьером безопасности (только UX)              │
└─────────────────────────────────────────────────────────────────┘
```

### 4.5 SQL-функция проверки прав

```sql
CREATE OR REPLACE FUNCTION public.has_permission(_permission_name text)
RETURNS boolean AS $$
DECLARE
  _user_id uuid;
  _has_perm boolean;
BEGIN
  _user_id := auth.uid();
  IF _user_id IS NULL THEN RETURN false; END IF;
  
  SELECT EXISTS (
    SELECT 1 FROM role_permissions rp
    JOIN user_roles ur ON ur.role = rp.role
    JOIN permissions p ON p.id = rp.permission_id
    WHERE ur.user_id = _user_id AND p.name = _permission_name
  ) INTO _has_perm;
  
  RETURN _has_perm;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

### 4.6 Критические инварианты

1. **Источник истины** — RLS policies в БД, не фронтенд
2. Проверки во фронтенде — только для UX (скрытие кнопок)
3. Edge Functions проверяют права ДО выполнения операций

### 7.3 Автозакрытие этапов

**Механизм**: pg_cron job каждые 5 минут

```sql
SELECT cron.schedule(
  'finalize-expired-stages',
  '*/5 * * * *',
  $$SELECT public.check_and_finalize_expired_stages()$$
);
```

**Функция finalize_expired_stage(p_stage_id):**

```sql
BEGIN
  -- 1. Деактивация parent_stage
  UPDATE parent_stages SET is_active = false WHERE id = p_stage_id;
  -- 2. Деактивация diagnostic_stages
  UPDATE diagnostic_stages SET is_active = false WHERE parent_id = p_stage_id;
  -- 3. Обновление meeting_stages
  UPDATE meeting_stages SET updated_at = NOW() WHERE parent_id = p_stage_id;
  -- 4. Expire assignments
  UPDATE survey_360_assignments SET status = 'expired' WHERE ...;
  -- 5. Expire tasks
  UPDATE tasks SET status = 'expired' WHERE ...;
  -- 6. Snapshot meetings
  UPDATE one_on_one_meetings SET status_at_stage_end = status, status = 'expired' WHERE ...;
END;
```

## 8. Диагностика 360 (end-to-end)

### 8.1 Полный флоу

```
1. СОЗДАНИЕ ЭТАПА
   Admin/HR → StagesPage → создаёт parent_stage + diagnostic_stage
   
2. ДОБАВЛЕНИЕ УЧАСТНИКОВ
   Admin/HR → addParticipants() → INSERT diagnostic_stage_participants
   → TRIGGER handle_diagnostic_participant_added
     → создаёт survey_360_assignments (self, manager)
     → создаёт tasks (peer_selection, self_assessment, manager_assessment)

3. ВЫБОР РЕСПОНДЕНТОВ (peer_selection)
   Employee → TasksManager → PeerSelectionButton → ColleagueSelectionDialog
   → создаёт survey_360_assignments (peer, status=pending)
   → завершает task (peer_selection → completed)
   → вызывает Edge Function create-peer-approval-task

4. УТВЕРЖДЕНИЕ РЕСПОНДЕНТОВ (peer_approval)
   Manager → ManagerRespondentApproval → approve/reject assignments
   → UPDATE survey_360_assignments SET status = 'approved' | 'rejected'
   → вызывает Edge Function create-peer-evaluation-tasks

5. ЗАПОЛНЕНИЕ ОПРОСОВ
   Any evaluator → UnifiedAssessmentPage
   → автосохранение с is_draft=true
   → финальное сохранение: is_draft=false

6. ПРОСМОТР РЕЗУЛЬТАТОВ + ОКНО ДЖОХАРИ
   Employee/Manager/HR → AssessmentResultsPage
   → useCorrectAssessmentResults(userId, filterType, stageId)
```

## 9. Окно Джохари (AI-анализ)

### 9.1 Назначение

Модуль AI-анализа зон восприятия soft skills по модели Окна Джохари. Анализирует расхождения между самооценкой и оценками окружающих, определяя навыки в одну из четырёх зон.

### 9.2 Зоны Джохари

| Зона | Ключ | Условие | Описание |
|------|------|---------|----------|
| Открытая зона | `arena` | `|self - others| < t_arena` | Навыки, одинаково воспринимаемые самим и окружающими |
| Слепая зона | `blind_spot` | `self > others` и `delta ≥ t_hi` | Самооценка завышена относительно окружающих |
| Скрытая зона | `hidden_strength` | `self < others` и `delta ≥ t_hi` | Самооценка занижена (скрытые сильные стороны) |
| Чёрный ящик | `unknown` | Недостаточно данных | Нет данных для классификации |

### 9.3 Два режима (scope)

| Scope | Описание | Кто в «others» | Мин. респондентов |
|-------|----------|----------------|-------------------|
| `all` | Все респонденты | manager + все peers | ≥ 3 |
| `external_only` | Только внешние | Только peers из категорий «(внешний)» | ≥ 3 внешних |

### 9.4 Пороги и формулы

```typescript
const scaleRange = scaleMax - scaleMin; // обычно 0-5
const tArena = 0.125 * scaleRange;       // порог «Открытая зона»
const tHi = 0.15 * scaleRange;           // порог Blind/Hidden

// Серая зона: tArena ≤ delta < tHi (относится к arena, но помечается grey_zone=true)

// Поляризация: среди others ≥ 3 респондентов, есть и оценки ≤ 33% шкалы и ≥ 67% шкалы
```

### 9.5 AI-генерация отчёта

**Edge Function**: `generate-johari-report`

**Входные параметры:**
```json
{
  "stage_id": "uuid",
  "evaluated_user_id": "uuid",
  "force_regenerate": false,
  "respondent_scope": "all" | "external_only"
}
```

**Логика:**
1. Проверка прав: `assessment_results.view_all` ИЛИ `assessment_results.view_team` ИЛИ manager оцениваемого
2. Загрузка `soft_skill_results` (is_draft=false, is_skip=false)
3. Определение типов оценщиков через `survey_360_assignments`
4. Определение внешних peers через `position_categories` (name содержит `(внешний)`)
5. Расчёт метрик по каждому навыку → распределение по зонам
6. Хэширование входных данных (SHA-256) → сравнение с существующим снимком
7. Если хэш совпадает и `force_regenerate=false` → возврат существующего снимка
8. Если данные изменились → AI-генерация (Gemini 2.5 Flash) → сохранение в `johari_ai_snapshots`
9. Для `external_only`: дополнительная AI-генерация case-based ревью комментариев

### 9.6 Case-based ревью комментариев (external_only)

**Назначение:** Анализ комментариев внешних респондентов в формате кейсов (не привязанных жёстко к одному навыку).

**Формат данных:**

```typescript
interface ExternalCommentsReview {
  summary_cases: SummaryCase[];
  one_to_one_questions: OneToOneQuestion[];
  notes: { comments_used: number };
}

interface SummaryCase {
  id: string;                // "case_1", "case_2", ...
  type: 'strength' | 'attention';
  title: string;             // краткий заголовок кейса
  insight: string;           // 1-2 предложения
  evidence_count: number;    // сколько комментариев подтверждают
  example_signals: string[]; // сигналы из комментариев
  related_zones: ('open' | 'blind' | 'hidden' | 'unknown')[];
  related_skills: { skill_name: string; relation: 'primary' | 'secondary' }[];
}

interface OneToOneQuestion {
  case_id: string;           // ссылка на кейс
  zone: 'open' | 'blind' | 'hidden' | 'unknown';
  question: string;          // вопрос для обсуждения на 1:1
}
```

**Правила генерации:**
1. Основная сущность — кейс, не навык
2. Каждый кейс подтверждён ≥ 1 комментарием
3. `related_skills` — не более 2 навыков, только как гипотеза; если связь слабая — пустой массив
4. Вопросы 1:1 формируются от кейсов
5. Тон нейтрально-деловой, без деанонимизации
6. При 0 комментариев — AI не вызывается, показывается fallback

**Где сохраняется:** `johari_ai_snapshots.external_comments_review` (JSONB)

## 10. Встречи 1:1 (end-to-end)

### 10.1 Полный флоу

```
1. СОЗДАНИЕ ЭТАПА ВСТРЕЧ
   Admin/HR → создаёт meeting_stage привязанный к parent_stage

2. ДОБАВЛЕНИЕ УЧАСТНИКОВ
   Admin/HR → addParticipants() → INSERT meeting_stage_participants
   → TRIGGER создаёт one_on_one_meetings со статусом 'draft'

3. ЗАПОЛНЕНИЕ ФОРМЫ (сотрудник)
   Employee → MeetingsPage → MeetingForm
   → поля: goal_and_agenda, energy_gained, energy_lost, stoppers

4. ОТПРАВКА НА УТВЕРЖДЕНИЕ
   Employee → submitMeeting() → status = 'submitted'

5. РАССМОТРЕНИЕ РУКОВОДИТЕЛЕМ
   Manager → approve / return

6. ДОБАВЛЕНИЕ РЕШЕНИЙ
   Both → MeetingDecisions → INSERT meeting_decisions
```

## 20. SQL-диагностики (read-only)

### 20.1 Активные этапы

```sql
SELECT ps.id, ps.period, ps.start_date, ps.end_date, ps.is_active,
       ds.id as diagnostic_stage_id, ds.is_active as ds_active,
       ms.id as meeting_stage_id
FROM parent_stages ps
LEFT JOIN diagnostic_stages ds ON ds.parent_id = ps.id
LEFT JOIN meeting_stages ms ON ms.parent_id = ps.id
WHERE ps.is_active = true
ORDER BY ps.start_date DESC;
```

```sql
SELECT ds.id as stage_id, a.status, a.assignment_type, COUNT(*) as count
FROM survey_360_assignments a

WHERE ds.is_active = true
GROUP BY ds.id, a.status, a.assignment_type;
```
