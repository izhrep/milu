## 2. Система авторизации

### 2.1 Архитектура

| Компонент | Путь | Назначение |
|-----------|------|------------|
| `AuthContext` | `src/contexts/AuthContext.tsx` | React Context для хранения состояния авторизации |
| `AuthGuard` | `src/components/AuthGuard.tsx` | HOC для защиты приватных маршрутов |
| `AuthPage` | `src/pages/AuthPage.tsx` | Страница входа |

### 2.2 AuthContext: структура состояния

```typescript
interface AuthContextType {
  user: AuthUser | null;      // Текущий пользователь
  logout: () => void;         // Функция выхода
  isAuthenticated: boolean;   // Флаг авторизации (!!user)
  loading: boolean;           // Флаг загрузки состояния
}

interface AuthUser {
  id: string;                 // UUID пользователя (= auth.users.id)
  full_name: string;          // ФИО (собирается из last_name, first_name, middle_name)
  email: string;              // Email
  role: string;               // Роль из user_roles.role
  first_name?: string;
  last_name?: string;
  middle_name?: string;
}
```

### 2.3 Жизненный цикл сессии

1. **Инициализация:** При монтировании `AuthProvider`:
   - Подписка на `supabase.auth.onAuthStateChange()`
   - Проверка существующей сессии через `supabase.auth.getSession()`
   
2. **Автообновление токена:** Supabase SDK автоматически обновляет access token при истечении (настройка `autoRefreshToken: true` в клиенте).

3. **Протухание сессии:** При истечении refresh token:
   - `onAuthStateChange` получает событие `SIGNED_OUT`
   - `user` устанавливается в `null`
   - AuthGuard перенаправляет на `/auth`

4. **Загрузка данных пользователя:** После получения сессии асинхронно загружается:
   - Роль из `user_roles` (запрос через Supabase)
   - Данные из `users` (first_name, last_name, middle_name)
   - Обратная совместимость: вызывается `decryptUserData()` (возвращает данные как есть)

### 2.4 Обработка ошибок

| Ошибка | Обработка |
|--------|-----------|
| Неверный логин/пароль | `toast.error('Ошибка входа: неверный Email или Пароль')` |
| Истёкшая сессия | Редирект на `/auth` |
| Ошибка загрузки роли | Устанавливается роль `employee` по умолчанию |

### 2.5 Маршрутизация

**Файл:** `src/App.tsx`

| Маршрут | Компонент | Защита |
|---------|-----------|--------|
| `/auth` | `AuthPage` | Публичный |
| `/*` (все остальные) | Обёрнуты в `AuthGuard` | Требуется авторизация |

**AuthGuard логика:**
```typescript
if (loading) → Показать спиннер
if (!isAuthenticated) → Navigate to /auth
else → Показать children
```

### 2.6 Cookie Consent

При входе проверяется `users.cookies_consent`:
- Если `false` — показывается чекбокс согласия
- Кнопка "Войти" недоступна до согласия
- При успешном входе обновляется `cookies_consent = true`, `cookies_consent_at = now()`

### 2.7 Хранение сессии

Supabase использует localStorage для хранения токенов:
- `sb-<project-ref>-auth-token` — JSON с access_token, refresh_token

## 3. Роли и разрешения

### 3.1 Список ролей

**Таблица:** `user_roles`  
**ENUM:** `app_role`

| Роль | Описание |
|------|----------|
| `admin` | Полный доступ ко всем функциям системы |
| `hr_bp` | HR Business Partner — управление диагностикой, этапами |
| `manager` | Руководитель — управление командой, утверждение респондентов |
| `employee` | Сотрудник — базовый доступ, прохождение оценок |

### 3.2 Система разрешений

**Таблицы:**
- `permissions` — справочник разрешений (name, resource, action)
- `role_permissions` — связь роль ↔ разрешение
- `permission_groups` — группировка разрешений для UI

**Функция проверки:** `has_permission(_permission_name text)` (SECURITY DEFINER)

### 3.3 Ключевые разрешения

| Разрешение | Ресурс | Действие | Роли |
|------------|--------|----------|------|
| `security.manage` | security | manage | admin |

| `users.manage_roles` | users | manage_roles | admin |
| `diagnostics.manage` | diagnostics | manage | admin, hr_bp |
| `diagnostics.view` | diagnostics | view | admin, hr_bp, manager |
| `diagnostics.manage_participants` | diagnostics | manage_participants | admin, hr_bp |
| `meetings.create` | meetings | create | admin, hr_bp |
| `meetings.view` | meetings | view | admin, hr_bp, manager |
| `development.manage` | development | manage | admin, hr_bp |
| `development.view_all` | development | view_all | admin, hr_bp |

### 3.4 Матрица доступа по экранам

| Экран/Маршрут | employee | manager | hr_bp | admin |

### 3.5 Источник истины

1. **RLS (Row Level Security)** — первичный барьер на уровне БД
2. **`has_permission()`** — проверка в RLS-политиках и триггерах
3. **UI-код** — проверка через `usePermission()` hook для скрытия элементов

**Хук для проверки:** `src/hooks/usePermission.ts`

```typescript
const { hasPermission, isLoading } = usePermission('security.manage');
const permissions = usePermissions(['users.create', 'diagnostics.view']);
const { isAdmin, isLoading } = useIsAdmin();
```

### 4.3 Ключевые инварианты

1. **Изоляция по diagnostic_stage_id:** Все результаты, задачи и assignments привязаны к конкретному этапу. Данные разных этапов не смешиваются.

2. **assignment ↔ task:** Каждый assignment может иметь связанную task через `tasks.assignment_id`. Задача создаётся триггером или Edge Function.

3. **Scoping:** Запросы результатов всегда фильтруют по `diagnostic_stage_id` для изоляции данных.

4. **Статусы assignments:** `pending` → `approved` → `completed` (или `rejected`, `expired`).

5. **is_draft:** Результаты с `is_draft = true` — черновики, не учитываются в агрегации.

### 4.4 Что НЕ хранится

- **PII шифрование:** Удалено, данные хранятся открыто
- **Файлы:** Система не использует Supabase Storage
- **История изменений результатов:** Хранится только последняя версия

## 5. Бизнес-логика диагностики

### 5.1 Типы оценки

| Тип | assignment_type | Описание |
|-----|-----------------|----------|
| Самооценка | `self` | Сотрудник оценивает себя |
| Оценка руководителя | `manager` | Руководитель оценивает подчинённого |
| Оценка коллеги | `peer` | Коллега оценивает сотрудника |

### 5.2 Структура этапов

```
parent_stage (Период Q1 2026)
├── diagnostic_stage (Диагностика компетенций)
│   ├── participants → self-assignments, manager-assignments
│   ├── peer-assignments (после утверждения)
│   └── tasks (peer_selection, peer_approval, survey_360_evaluation)
└── meeting_stage (Встречи 1:1)
    ├── participants
    └── one_on_one_meetings
```

### 5.3 Статусы назначений (survey_360_assignments)

| Статус | Описание |
|--------|----------|
| `pending` | Ожидает утверждения (для peer) или начала заполнения |
| `approved` | Утверждено руководителем (peer готов к оценке) |
| `completed` | Оценка завершена |
| `rejected` | Отклонено руководителем |
| `expired` | Истёк срок этапа |

### 5.4 Точки входа в коде

| Функциональность | Файл |
|------------------|------|
| Управление этапами | `src/components/UnifiedStagesManager.tsx` |



## 9. Расчет средних значений

### 9.1 Агрегация Soft Skills

**Хук:** `useSurvey360Results.ts`

**Алгоритм:**
1. Получить все `soft_skill_results` для `evaluated_user_id`
2. Исключить `is_skip = true`
3. Группировать по `quality_id` (soft skill)
4. Для каждой группы вычислить:
   - `self_score`: AVG где `evaluating_user_id = evaluated_user_id`
   - `supervisor_scores`: AVG где `evaluating_user_id = manager_id`
   - `colleague_scores`: AVG для остальных
5. `overall_average`: AVG всех оценок

```typescript
// Код из useSurvey360Results.ts
const selfAvg = data.self_scores.reduce((sum, score) => sum + score, 0) / data.self_scores.length;
const colleagueAvg = data.colleague_scores.reduce((sum, score) => sum + score, 0) / data.colleague_scores.length;
```

### 9.2 Агрегация Hard Skills

**Хук:** `useSkillSurveyResults.ts`

**Алгоритм:**
1. Получить все `hard_skill_results` для `evaluated_user_id`
2. Исключить `is_skip = true`
3. JOIN с `hard_skill_answer_options` для получения `numeric_value`
4. Группировать по `skill_id`
5. `average_score = SUM(numeric_value) / COUNT(*)`

### 9.3 Что исключается из расчёта

- Записи с `is_draft = true` — не финализированные ответы
- Записи с `is_skip = true` — пропущенные вопросы

### 12.1 Структура задачи


### 12.4 Фильтрация в UI

**Хук:** `useTasks.ts`

**Правила отображения в "Мои задачи":**
1. Только `status IN ('pending', 'in_progress')`

3. Исключаются задачи с `deadline < today`
4. Исключаются задачи со статусом `expired`

**Код:**
```typescript
const filteredTasks = allTasks.filter(task => {
  if (task.status === 'expired') return false;
  if (task.deadline && new Date(task.deadline) < today) return false;
  if (!task.diagnostic_stage_id) return true;
  return activeStageIds.includes(task.diagnostic_stage_id);
});
```

## 17. RLS и безопасность

### 17.1 Основные паттерны RLS

**Паттерн 1: Доступ к своим данным**
```sql
USING (user_id = auth.uid())
```

**Паттерн 2: Доступ через разрешения**
```sql
USING (has_permission('diagnostics.manage'))
```

**Паттерн 3: Доступ для руководителя**
```sql
USING (
  user_id = auth.uid() OR
  EXISTS (SELECT 1 FROM users WHERE id = <table>.user_id AND manager_id = auth.uid())
)
```

### 17.2 RLS по ключевым таблицам

| `survey_360_assignments` | Свои + менеджер + разрешение | Триггеры | Свои | — |

## 19. Автоматическое закрытие этапов

### 19.1 Архитектура pg_cron

**Задание:**
```sql
SELECT cron.schedule(
  'finalize-expired-stages',
  '*/5 * * * *',  -- каждые 5 минут
  $$SELECT public.check_and_finalize_expired_stages()$$
);
```

### 19.2 Функция check_and_finalize_expired_stages

```sql
FOR stage_record IN
  SELECT id FROM parent_stages
  WHERE is_active = true AND end_date < CURRENT_DATE
LOOP
  PERFORM finalize_expired_stage(stage_record.id);
END LOOP;
```

### 19.3 Функция finalize_expired_stage(p_stage_id)

1. `parent_stages.is_active = false`
2. `diagnostic_stages.is_active = false` WHERE `parent_id = p_stage_id`
3. `meeting_stages.updated_at = now()` WHERE `parent_id = p_stage_id`
4. Snapshot + `status = 'expired'` для:
   - `survey_360_assignments` (кроме completed, rejected, expired)
   - `tasks` (кроме completed, expired)
   - `one_on_one_meetings` (кроме approved, expired)

### 19.4 Снапшот статуса

| Поле | Назначение |
|------|------------|
| `status_at_stage_end` | Статус до закрытия |
| `stage_end_snapshot_at` | Время закрытия |

### 19.5 Переоткрытие этапа

```sql
SELECT reopen_expired_stage('<stage_id>');
```

Восстанавливает статусы из снапшотов.
