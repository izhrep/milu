## 🔧 Обнаруженные и исправленные проблемы

### 1. Структура данных и индексы

#### Проблемы:
- ❌ Отсутствовали индексы для частых запросов
- ❌ Не было ограничения на уникальность встреч (employee + stage)
- ❌ Не было проверки employee_id ≠ manager_id

#### Решение:
✅ Добавлены 6 новых индексов:
- `idx_one_on_one_meetings_employee` - для поиска встреч сотрудника
- `idx_one_on_one_meetings_manager` - для поиска встреч руководителя
- `idx_one_on_one_meetings_status` - для фильтрации по статусу
- `idx_meeting_decisions_meeting` - для быстрого доступа к решениям
- `idx_meeting_stage_participants_user` - для поиска участников
- `idx_tasks_meeting_type` - для фильтрации задач встреч

✅ Добавлены ограничения:
- `idx_one_meeting_per_employee_stage` (UNIQUE) - одна встреча на сотрудника в рамках этапа
- `check_employee_manager_different` (CHECK) - employee_id ≠ manager_id

### 2. Триггеры и функции

#### Проблемы:
- ❌ Дублирование задач при повторном добавлении участника
- ❌ Неправильная обработка статусов задач при возврате встречи
- ❌ Отсутствие автоматического создания встреч при добавлении участника
- ❌ Отсутствие функции проверки консистентности данных

#### Решение:
✅ **Создана функция `create_meeting_for_participant()`**
```sql
-- Автоматически создаёт one_on_one_meetings при добавлении участника
-- Проверяет наличие руководителя
-- Предотвращает дубликаты
```

✅ **Оптимизирована функция `create_meeting_task_for_participant()`**
```sql
-- Создаёт задачу только если её ещё нет
-- Использует SECURITY DEFINER и search_path = 'public'
-- Логирует действия через RAISE NOTICE
```

✅ **Оптимизирована функция `update_meeting_task_status()`**
```sql
-- При approved → задача становится completed
-- При returned → задача возвращается в pending
-- При submitted → задача остаётся pending
```

✅ **Создана функция `check_meetings_data_consistency()`**
```sql
-- Проверяет 5 типов несоответствий:
-- 1. Участники без встреч
-- 2. Встречи без участников
-- 3. Участники без задач
-- 4. Встречи с некорректными timestamp-полями
-- 5. Решения без встреч
```

✅ **Триггеры**
- `create_meeting_for_participant_trigger` (AFTER INSERT on meeting_stage_participants)
- `create_meeting_task_for_participant_trigger` (AFTER INSERT on meeting_stage_participants)
- `update_meeting_task_on_approval` (AFTER UPDATE OF status on one_on_one_meetings)

### 3. RLS политики безопасности

#### Проблемы:
- ❌ Слишком широкие политики на one_on_one_meetings
- ❌ Нет разделения прав для employee/manager
- ❌ Недостаточная детализация на meeting_decisions

#### Решение:
✅ **Переписаны политики для `one_on_one_meetings`**

| Операция | Роль | Условие |
|----------|------|---------|
| SELECT | Employee | `employee_id = current_user` |
| SELECT | Manager | `manager_id = current_user` |
| SELECT | HR/Admin | `is_current_user_hr()` OR `is_current_user_admin()` |
| INSERT | Employee | `employee_id = current_user` |
| UPDATE | Employee | Только в статусах `draft`, `returned`, `submitted` |
| UPDATE | Manager | Только в статусах `submitted`, `approved`, `returned` |
| DELETE | Admin | Только администраторы |

✅ **Переписаны политики для `meeting_decisions`**
- Доступ только участникам встречи (employee/manager)
- HR и Admin имеют полный доступ
- Проверка через JOIN с `one_on_one_meetings`

✅ **Сохранены политики для `meeting_stages` и `meeting_stage_participants`**
- Администраторы управляют этапами
- Руководители видят этапы своих подчинённых
- Участники видят свои этапы

### 4. Жизненный цикл встреч

#### Проблемы:
- ❌ Неявные переходы между статусами
- ❌ Отсутствие контроля за заполнением временных полей

#### Решение:
✅ **Чёткая модель статусов:**

```
draft → submitted → approved ✓
         ↓
       returned → submitted → approved ✓
```

✅ **Контроль временных полей:**
- При переходе в `submitted` → устанавливается `submitted_at`
- При переходе в `approved` → устанавливается `approved_at`
- При переходе в `returned` → устанавливается `returned_at`
- Триггер `update_updated_at_column` обновляет `updated_at`

### 5. Edge cases

#### Обработанные случаи:
✅ Сотрудник без руководителя
- Встреча не создаётся
- Логируется через RAISE NOTICE

✅ Смена руководителя
- Существующие встречи сохраняют старого руководителя
- Новые встречи создаются с новым руководителем

✅ Повторное добавление участника
- Предотвращается через UNIQUE constraint на (stage_id, user_id)
- Проверка на существование встречи и задачи

✅ Деактивация этапа (is_active = false)
- Встречи и задачи остаются в БД
- Фильтрация в UI по is_active

## 🎯 Финальный флоу встреч 1:1

### Шаг 1: Создание этапа
**Роль**: HR/Admin  
**Действие**: Создание записи в `meeting_stages`
- Указывается период (например, "Q1 2025")
- Устанавливаются даты: start_date, end_date, deadline_date
- is_active = true

### Шаг 2: Добавление участников
**Роль**: HR/Admin  
**Действие**: Добавление записей в `meeting_stage_participants`

**Автоматические действия** (триггеры):
1. ✅ Создаётся `one_on_one_meetings` (если есть руководитель)
   - employee_id = участник
   - manager_id = руководитель участника
   - status = 'draft'
   
2. ✅ Создаётся `tasks`
   - user_id = участник
   - task_type = 'meeting'
   - category = 'Встречи 1:1'
   - title = "Встреча 1:1 - [период]"
   - status = 'pending'
   - deadline = deadline_date из этапа

### Шаг 3: Заполнение формы встречи
**Роль**: Сотрудник (Employee)  
**Статус встречи**: `draft`

Сотрудник заполняет поля:
- meeting_date
- goal_and_agenda
- energy_gained
- energy_lost
- stoppers
- previous_decisions_debrief (если есть предыдущие встречи)

Добавляет решения в `meeting_decisions`:
- decision_text
- is_completed = false

### Шаг 4: Отправка на утверждение
**Роль**: Сотрудник  
**Действие**: Изменение status → `submitted`

**Автоматические действия**:
- ✅ Устанавливается submitted_at = now()
- ✅ updated_at = now()
- ⚠️ Задача остаётся pending (ждёт утверждения)

### Шаг 5: Рассмотрение руководителем
**Роль**: Руководитель (Manager)

#### Вариант А: Утверждение
**Действие**: Изменение status → `approved`
- Заполняется manager_comment (опционально)

**Автоматические действия**:
- ✅ Устанавливается approved_at = now()
- ✅ Задача встречи → status = 'completed'
- ✅ updated_at = now()

#### Вариант Б: Возврат на доработку
**Действие**: Изменение status → `returned`
- Обязательно заполняется return_reason

**Автоматические действия**:
- ✅ Устанавливается returned_at = now()
- ✅ Задача → status = 'pending' (возврат к сотруднику)
- ✅ updated_at = now()

### Шаг 6: Доработка и повторная отправка
**Роль**: Сотрудник  
**Статус**: `returned` → `submitted` → `approved`

Повторяется цикл с Шага 3, пока встреча не будет утверждена.

### Шаг 7: Работа с решениями
**Роль**: Сотрудник/Руководитель  
**После утверждения**:

- Решения (`meeting_decisions`) доступны для просмотра
- is_completed можно обновлять
- Решения используются для:
  - Контроля выполнения договорённостей
  - Создания задач развития (в будущем)
  - Дебрифа на следующей встрече (previous_decisions_debrief)

- [ ] НЕ видит встречи других сотрудников

- [ ] Видит встречи своих подчинённых (manager_id = current_user)

- [ ] НЕ видит встречи чужих подчинённых
- [ ] НЕ может удалять встречи

- [ ] Видит все встречи (только чтение)
- [ ] Может управлять этапами
- [ ] Может добавлять участников

#### Admin
- [ ] Полный доступ ко всем встречам
- [ ] Может удалять встречи
