## 1. ОБЩЕЕ ОПИСАНИЕ ПРОЕКТА

### 1.1 Назначение системы

Комплексная веб-система для управления человеческими ресурсами с фокусом на:

- **Комплексную оценку компетенций** — объединённая оценка 360° (soft skills) + профессиональные навыки (hard skills)
- **Карьерное развитие** — треки роста, планы развития, задачи
- **Встречи 1:1** — периодические встречи сотрудников с руководителями
- **Периодические диагностические циклы** — H1/H2 оценка с автоматизацией задач
- **HR-аналитику** — отчёты, мониторинг прогресса, gap-анализ

### 1.2 Ключевые бизнес-процессы

1. **Диагностический цикл:**
   - HR создаёт этап диагностики (H1_2025, H2_2025)
   - Добавляет участников → автоматически создаются задачи и назначения
   - Сотрудники проходят объединённую оценку (360 + навыки)
   - Результаты агрегируются в `user_assessment_results`
   - GAP-анализ генерирует рекомендации по развитию

2. **Встречи 1:1:**
   - HR создаёт этап встреч
   - Добавляет участников → автоматически создаются задачи
   - Сотрудник заполняет форму → отправляет на утверждение
   - Руководитель проверяет → утверждает или возвращает
   - При утверждении задача автоматически закрывается

3. **Карьерное развитие:**
   - На основе результатов диагностики создаются планы развития
   - Генерируются задачи для устранения разрывов компетенций
   - Отслеживается прогресс по карьерным трекам

### 3.3 Использование авторизации на фронте

**AuthContext** используется во всех защищённых компонентах:

```tsx
const { user, isAuthenticated, loading, logout } = useAuth();

if (!isAuthenticated) {
  // Redirect via AuthGuard
}

// user.id → текущий пользователь
// user.role → роль (admin, hr_bp, manager, employee)
```

**Проверка прав через `usePermission` хук:**

```tsx
const { hasPermission } = usePermission();

if (hasPermission('diagnostics.create')) {
  // Показать кнопку создания диагностики
}
```

**Условный рендеринг по ролям:**

```tsx
{user.role === 'admin' && (
  <AdminOnlyButton />
)}
```

### 3.4 Привязка фронта к API

**Используемые подходы:**

1. **Supabase Client (`supabase.from('table').select()`):**
   - Большинство запросов идёт через Supabase JS Client
   - Автоматическая обработка JWT токенов
   - RLS политики применяются автоматически

2. **RPC вызовы (`supabase.rpc('function_name', params)`):**
   - Для сложной логики (агрегация, расчёты)
   - Примеры: `refresh_user_effective_permissions`, `calculate_diagnostic_stage_progress`

3. **Edge Functions (`supabase.functions.invoke('function-name')`):**
   - Для операций требующих серверную логику
   - Примеры: `create-user`, `delete-user`, `create-peer-evaluation-tasks`

4. **Внешние API (Yandex Cloud Functions):**
   - Шифрование/дешифрование данных пользователей
   - Вызывается через `fetch()` из хуков (`useUsers`)

**Примеры из ключевых страниц:**

- **`MyAssignmentsPage`**: `supabase.from('survey_360_assignments').select()`
- **`DiagnosticMonitoringPage`**: `supabase.rpc('calculate_diagnostic_stage_progress')`
- **`UnifiedAssessmentPage`**: `supabase.from('soft_skill_questions').select()`, `supabase.from('hard_skill_results').insert()`

### 4.1 Supabase Edge Functions

**Расположение:** `supabase/functions/`

#### 4.1.1 `create-user/index.ts`

**Назначение:** Создание нового пользователя в системе.

**Метод:** `POST`

**Параметры:**
- `email` (string)
- `password` (string)
- `first_name` (string)
- `last_name` (string)
- `middle_name` (string, optional)
- `role` (string: admin, hr_bp, manager, employee)
- `position_id`, `department_id`, `grade_id` (uuid, optional)

**Логика:**
1. Проверяет JWT токен и право `users.create`
2. Создаёт пользователя в `auth.users` через `supabaseAdmin.auth.admin.createUser()`
3. Вставляет запись в `users` с `id = auth_user.id`
4. Вставляет роль в `user_roles`
5. Вызывает `refresh_user_effective_permissions()`
6. В случае ошибки — откатывает создание

**Возвращает:** `{ success: true, user }` или `{ error: message }`

**Вызывается из:** `CreateUserPage`

#### 4.1.2 `delete-user/index.ts`

**Назначение:** Удаление пользователя.

**Метод:** `POST`

**Параметры:**
- `userId` (uuid)
- `adminId` (uuid)

**Логика:**
1. Проверяет JWT токен и право `users.delete`
2. Получает данные пользователя для логирования
3. Дешифрует данные через Yandex Cloud Functions
4. Удаляет связанные записи (`audit_log`, `admin_activity_logs`, `user_roles`)
5. Логирует действие через `log_admin_action` RPC
6. Удаляет из `users` (каскадно удаляет связанные данные)
7. Удаляет из `auth.users`

**Возвращает:** `{ success: true }` или `{ error: message }`

**Вызывается из:** `UsersListPage`

#### 4.1.3 `create-peer-evaluation-tasks/index.ts`

**Назначение:** Создание задач для peer-evaluation после утверждения коллег.

**Метод:** `POST`

**Параметры:**
- `approvedAssignments` (array of { evaluating_user_id, assignment_id })
- `diagnosticStageId` (uuid)
- `evaluatedUserName` (string)
- `deadline` (date)

**Логика:**
1. Проверяет JWT токен
2. Для каждого `approvedAssignment`:
   - Проверяет существование задачи
   - Создаёт задачу если её нет:
     - `title` = "Оценить коллегу: [ФИО]"
     - `task_type` = 'peer_evaluation'
     - `status` = 'pending'

**Возвращает:** `{ success: true, tasksCreated: count }`

**Вызывается из:** `RespondentApprovalDialog` после утверждения коллег

#### 4.1.4 `generate-development-tasks/index.ts`

**Назначение:** Генерация задач развития на основе GAP-анализа.

### 4.2 PostgreSQL RPC функции

**Расположение:** определены в миграциях `supabase/migrations/`

#### 4.2.1 Авторизация и права

**`has_permission(_user_id uuid, _permission_name text) → boolean`**

**Назначение:** Проверяет, есть ли у пользователя конкретное право.

**Реализация:**
```sql
CREATE OR REPLACE FUNCTION has_permission(_user_id uuid, _permission_name text)
RETURNS boolean
LANGUAGE sql STABLE SECURITY DEFINER
SET search_path = public
AS $$
  SELECT EXISTS (
    SELECT 1
    FROM user_roles ur
    JOIN role_permissions rp ON rp.role = ur.role
    JOIN permissions p ON p.id = rp.permission_id
    WHERE ur.user_id = _user_id
      AND p.name = _permission_name
  );
$$;
```

**Используется в:**
- RLS политиках (`USING (has_permission(auth.uid(), 'users.view'))`)
- Frontend через хук `usePermission`

**`refresh_user_effective_permissions(target_user_id uuid) → void`**

**Назначение:** Обновляет кэш эффективных прав пользователя.

**Логика:**
1. Удаляет старые записи из `user_effective_permissions`
2. Вставляет актуальные права на основе `user_roles` + `role_permissions`

**Вызывается:**
- Триггерами при изменении `user_roles` или `role_permissions`
- Вручную при создании пользователя

**`is_users_manager(target_user_id uuid) → boolean`**

**Назначение:** Проверяет, является ли текущий пользователь руководителем указанного.

**Используется в:** RLS политиках для доступа managers к данным подчинённых

#### 4.2.2 Диагностика

**`calculate_diagnostic_stage_progress(stage_id uuid) → numeric`**

**Назначение:** Вычисляет процент завершения этапа диагностики.

**Логика:**
- Считает количество участников
- Считает количество завершённых оценок
- Возвращает процент

**Вызывается:** Автоматически триггером при сохранении результатов

**`is_diagnostic_stage_participant(_stage_id uuid, _user_id uuid) → boolean`**

**Назначение:** Проверяет, является ли пользователь участником этапа.

**Используется в:** RLS политиках для доступа к данным диагностики

#### 4.2.3 Оценка и результаты

**`aggregate_survey_360_results() → trigger`**

**Назначение:** Агрегирует результаты оценки 360 в `user_assessment_results`.

**Логика:**
1. Триггер срабатывает после INSERT в `soft_skill_results` с `is_draft=false`
2. Удаляет старые результаты для данного пользователя + период + качество
3. Группирует ответы по `quality_id`:
   - `self_assessment` — среднее где `evaluating_user_id = evaluated_user_id`
   - `peers_average` — среднее где `evaluating_user_id ≠ evaluated_user_id AND ≠ manager_id`
   - `manager_assessment` — среднее где `evaluating_user_id = manager_id`
4. Вставляет в `user_assessment_results`

**`aggregate_skill_survey_results() → trigger`**

**Назначение:** Агрегирует результаты оценки навыков (аналогично 360).

**Логика:** Та же, но для `hard_skill_results` и `skill_id`

#### 4.2.4 Задачи

**`log_admin_action(action_type text, details jsonb) → void`**

**Назначение:** Логирование административных действий.

**Вставляет в:** `admin_activity_logs`

**Вызывается из:** Edge functions при создании/удалении пользователей

## 6. ТРИГГЕРЫ И ФОНОВЫЕ ЗАДАЧИ

### 6.1 Триггеры на таблице `users`

**`update_users_updated_at`**
- **Событие:** BEFORE UPDATE
- **Функция:** `update_updated_at_column()`
- **Действие:** Обновляет `updated_at = now()`

### 6.2 Триггеры на таблице `user_roles`

**`trigger_refresh_user_permissions`**
- **Событие:** AFTER INSERT OR UPDATE OR DELETE
- **Функция:** `refresh_user_permissions_trigger()`
- **Действие:** Вызывает `refresh_user_effective_permissions(NEW.user_id или OLD.user_id)`

### 6.3 Триггеры на таблице `role_permissions`

**`trigger_refresh_role_permissions`**
- **Событие:** AFTER INSERT OR UPDATE OR DELETE
- **Функция:** `refresh_all_users_with_role_permissions()`
- **Действие:** Обновляет `user_effective_permissions` для всех пользователей с изменённой ролью

### 6.4 Триггеры на таблице `diagnostic_stage_participants`

**`assign_surveys_to_diagnostic_participant`**
- **Событие:** AFTER INSERT
- **Функция:** `auto_assign_surveys_and_create_tasks()`
- **Действие:**
  1. Создаёт `survey_360_assignments`:
     - Самооценка: `evaluated_user_id = user_id`, `evaluating_user_id = user_id`, `status = 'approved'`
     - Оценка руководителем: `evaluated_user_id = user_id`, `evaluating_user_id = manager_id`, `status = 'approved'`, `is_manager_participant = true`
  2. Создаёт задачи для самооценки и оценки навыков

**`create_diagnostic_task_for_participant`**
- **Событие:** AFTER INSERT
- **Функция:** `create_diagnostic_task()`
- **Действие:** Создаёт задачу типа `diagnostic_stage` с дедлайном из `diagnostic_stages.deadline_date`

### 6.5 Триггеры на таблице `soft_skill_results`

**`set_evaluation_period`**
- **Событие:** BEFORE INSERT
- **Функция:** `auto_set_evaluation_period()`
- **Действие:** Устанавливает `evaluation_period = H1_YYYY или H2_YYYY` на основе `created_at`

**`aggregate_survey_360_results`**
- **Событие:** AFTER INSERT OR UPDATE
- **Условие:** `NEW.is_draft = false`
- **Функция:** `aggregate_soft_skill_results()`
- **Действие:**
  1. Удаляет старые результаты из `user_assessment_results` для данного пользователя + период + качество
  2. Группирует ответы по `quality_id`
  3. Вычисляет средние:
     - `self_assessment` — где `evaluating_user_id = evaluated_user_id`
     - `peers_average` — где `evaluating_user_id ≠ evaluated_user_id AND ≠ manager_id`
     - `manager_assessment` — где `evaluating_user_id = manager_id`
  4. Вставляет агрегированные данные в `user_assessment_results`

**`update_diagnostic_stage_status`**
- **Событие:** AFTER INSERT OR UPDATE
- **Функция:** `update_diagnostic_progress()`
- **Действие:** Вызывает `calculate_diagnostic_stage_progress(diagnostic_stage_id)` и обновляет `progress_percent`

### 6.6 Триггеры на таблице `hard_skill_results`

- `aggregate_skill_survey_results`

### 7.2 Получение текущего пользователя на фронте

### 7.5 Ограничение доступа к данным через RLS

**Все таблицы с пользовательскими данными имеют RLS включённый:**

```sql
ALTER TABLE users ENABLE ROW LEVEL SECURITY;
ALTER TABLE tasks ENABLE ROW LEVEL SECURITY;
ALTER TABLE diagnostic_stages ENABLE ROW LEVEL SECURITY;
-- и т.д.
```

**Пример RLS политики:**
```sql
CREATE POLICY "users_select_policy" ON users
FOR SELECT
USING (
  id = get_current_user_id()  -- Свои данные
  OR has_permission(get_current_user_id(), 'users.view')  -- Или есть право
  OR is_users_manager(id)  -- Или руководитель
);
```

**Таким образом:**
- Пользователь может видеть только:
  - Свои данные
  - Данные, разрешённые его permissions
  - Данные подчинённых (если он manager)
- Admin видит всё (через `has_permission('users.view')`)

### 8.4 Логика проверки прав

**Схема проверки:**
```
1. Запрос → Supabase → RLS Policy
2. RLS Policy → has_permission(auth.uid(), 'permission.name')
3. has_permission → user_effective_permissions
4. user_effective_permissions → TRUE/FALSE
5. TRUE → Запрос выполнен
6. FALSE → Access Denied (403)
```

**Кэширование:**
- `user_effective_permissions` — материализованная таблица
- Обновляется автоматически триггерами при изменении `user_roles` или `role_permissions`
- Обеспечивает быструю проверку прав без join'ов

## 9. БИЗНЕС-ЛОГИКА КЛЮЧЕВЫХ МОДУЛЕЙ

### 9.1 Модуль "Диагностика компетенций"

#### 9.1.1 Основные сценарии использования

**1. Создание диагностического этапа (HR_BP/Admin):**
- Переход на `/admin/diagnostics`
- Создание этапа с указанием:
  - Период (H1_2025, H2_2025)
  - Даты: `start_date`, `end_date`, `deadline_date`
- Автоматическое определение `evaluation_period` (H1_YYYY или H2_YYYY)
- Статус устанавливается в `setup`

**2. Добавление участников (HR_BP/Admin):**
- Выбор пользователей из списка
- При добавлении в `diagnostic_stage_participants` триггеры автоматически:
  - Создают `survey_360_assignments` для самооценки (status = approved)
  - Создают `survey_360_assignments` для оценки руководителем (status = approved, is_manager_participant = true)
  - Создают задачу типа `diagnostic_stage` с дедлайном

**3. Прохождение оценки (Employee):**
- Пользователь видит задачу в разделе "Задачи"
- Кликает "Пройти оценку" → открывается `/unified-assessment/:assignmentId`
- `UnifiedAssessmentPage`:
  - Загружает вопросы по `soft_skill_questions` (фильтр по `quality_id` из `grade_qualities`)
  - Загружает вопросы по `hard_skill_questions` (фильтр по `skill_id` из `grade_skills`)
  - Показывает табы "Soft Skills" и "Hard Skills"
  - Автосохранение черновиков (`is_draft=true`)
  - Финальная отправка (`is_draft=false`)

**4. Агрегация результатов (автоматически):**
- Триггер `aggregate_survey_360_results` срабатывает при `is_draft=false`
- Группирует ответы по `quality_id` или `skill_id`
- Вычисляет средние по типам оценивающих:
  - `self_assessment` — самооценка
  - `peers_average` — среднее по коллегам
  - `manager_assessment` — оценка руководителя
- Вставляет в `user_assessment_results`

**5. Просмотр результатов (Employee/Manager/HR):**
- Переход на `/assessment/results/:userId`
- `AssessmentResultsPage` отображает:
  - Роза компетенций (radar chart)
  - Таблица с разбивкой по навыкам/качествам
  - GAP-анализ (целевой уровень vs текущий)
  - Рекомендации по развитию

#### 9.1.2 Используемые таблицы

- `diagnostic_stages` — этапы диагностики
- `diagnostic_stage_participants` — участники
- `survey_360_assignments` — назначения оценок
- `soft_skill_questions`, `hard_skill_questions` — вопросы
- `soft_skill_results`, `hard_skill_results` — ответы
- `user_assessment_results` — агрегированные результаты
- `tasks` — задачи для прохождения оценки

#### 9.1.3 Ключевые API вызовы

**Создание этапа:**
```tsx
await supabase.from('diagnostic_stages').insert({
  period: 'H1_2025',
  start_date: '2025-01-01',
  end_date: '2025-06-30',

  created_by: currentUserId
});
```

**Добавление участника:**
```tsx
await supabase.from('diagnostic_stage_participants').insert({
  stage_id: stageId,

```

**Загрузка вопросов:**

const { data: softSkillQuestions } = await supabase

  .in('quality_id', qualityIds); // qualityIds из grade_qualities

await supabase.from('soft_skill_results').insert({

#### 9.1.4 Триггеры и автоматизация

1. **При добавлении участника:**
   - `assign_surveys_to_diagnostic_participant` → создаёт assignments
   - `create_diagnostic_task_for_participant` → создаёт задачу

2. **При сохранении ответа (`is_draft=false`):**
   - `set_evaluation_period` → устанавливает H1/H2
   - `aggregate_survey_360_results` → агрегирует в `user_assessment_results`
   - `update_diagnostic_stage_status` → обновляет прогресс этапа

3. **При завершении всех вопросов для assignment:**
   - `update_assignment_on_survey_completion` → меняет статус на "выполнено"
   - Задача автоматически закрывается

#### 9.1.5 Статусы диагностического этапа

| Статус | Описание |
|--------|----------|
| `setup` | Настройка (добавление участников) |
| `assessment` | Идёт оценка |
| `completed` | Завершено |

**Переход между статусами:**
- `setup` → `assessment` — вручную HR_BP
- `assessment` → `completed` — автоматически при 100% прогрессе

- Создание этапа:
  - Период (Q1_2025)
  - Даты: `start_date`, `end_date`, `deadline_date`

**2. Добавление участников (HR_BP/Admin):**
- Выбор пользователей
- При добавлении в `meeting_stage_participants` триггер:
  - Создаёт задачу типа `meeting` с дедлайном

**3. Заполнение формы встречи (Employee):**
- Переход на `/meetings`
- Создание новой встречи
- Заполнение полей:
  - `goal_and_agenda` — цель и повестка
  - `energy_gained` — откуда энергия
  - `energy_lost` — куда уходит энергия
  - `stoppers` — что мешает
  - `previous_decisions_debrief` — дебриф прошлых решений
- Добавление решений (`meeting_decisions`) — минимум 1
- Отправка на утверждение → `status = submitted`

**4. Рассмотрение формы (Manager):**
- Вкладка "Встречи подчинённых"
- Просмотр всех полей
- Добавление `manager_comment`
- Редактирование решений
- Действия:
  - **Утвердить** → `status = approved`, `approved_at = now()`
  - **Вернуть на доработку** → `status = returned`, `return_reason` указывается

**5. Возврат на доработку (если returned):**
- Сотрудник видит комментарий руководителя и `return_reason`
- Вносит правки
- Повторно отправляет → `status = submitted`

**6. Утверждение (approved):**
- Триггер `close_meeting_task_on_approval` закрывает связанную задачу
- Возможность создать новую встречу в рамках этапа

#### 9.2.2 Используемые таблицы

- `meeting_stages` — этапы встреч
- `meeting_stage_participants` — участники
- `one_on_one_meetings` — встречи
- `meeting_decisions` — решения
- `tasks` — задачи на проведение встречи

#### 9.2.3 Ключевые API вызовы

**Создание встречи:**
```tsx
await supabase.from('one_on_one_meetings').insert({
  stage_id: stageId,
  employee_id: userId,
  manager_id: managerId,
  status: 'draft',
  goal_and_agenda: text,
  energy_gained: text,
  energy_lost: text,
  stoppers: text
});
```

**Добавление решений:**
```tsx
await supabase.from('meeting_decisions').insert({
  meeting_id: meetingId,
  decision_text: text,
  created_by: userId
});
```

**Отправка на утверждение:**
```tsx
await supabase.from('one_on_one_meetings')
  .update({
    status: 'submitted',
    submitted_at: new Date().toISOString()
  })
  .eq('id', meetingId);
```

**Утверждение (Manager):**
```tsx
await supabase.from('one_on_one_meetings')
  .update({
    status: 'approved',
    approved_at: new Date().toISOString(),
    manager_comment: comment
  })
  .eq('id', meetingId);
// → Триггер close_meeting_task_on_approval автоматически закрывает задачу
```

**Возврат на доработку:**
```tsx
await supabase.from('one_on_one_meetings')
  .update({
    status: 'returned',
    returned_at: new Date().toISOString(),
    return_reason: reason
  })
  .eq('id', meetingId);
```

#### 9.2.4 Триггеры и автоматизация

1. **При добавлении участника:**
   - `create_meeting_task_for_participant` → создаёт задачу

2. **При утверждении встречи:**
   - `close_meeting_task_on_approval` → закрывает задачу (`status = completed`)

#### 9.2.5 Статусы встречи

| Статус | Описание |
|--------|----------|
| `draft` | Черновик (заполняется сотрудником) |
| `submitted` | Отправлено на утверждение |
| `returned` | Возвращено на доработку |
| `approved` | Утверждено руководителем |

**3. Создание плана развития (Manager/HR):**

**4. Выбор карьерного трека (Employee):**
- Просмотр доступных треков
- Выбор трека → создание `user_career_progress`

#### 9.3.2 Используемые таблицы

## 10. ИЗВЕСТНЫЕ ОГРАНИЧЕНИЯ И ОСОБЕННОСТИ

### 10.1 Шифрование данных пользователей

**Что зашифровано:**
- `users.first_name`
- `users.last_name`
- `users.middle_name`
- `users.email`

**Как работает:**
- Данные хранятся в БД в зашифрованном виде (AES-256)
- Дешифрование происходит через Yandex Cloud Functions
- API URL: `https://functions.yandexcloud.net/d4eb74i8p2s72d275h1g`

**Логика:**
```tsx
// useUsers.ts
const decryptResponse = await fetch(API_URL, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    action: 'decrypt',
    data: {
      first_name: user.first_name,
      last_name: user.last_name,
      middle_name: user.middle_name,
      email: user.email
    }
  })
});

const decrypted = await decryptResponse.json();
// → { first_name: "Иван", last_name: "Иванов", ... }
```

**Ограничение:**
- Дешифрование происходит на клиенте (медленно для больших списков)
- Невозможен поиск по зашифрованным полям в БД

### 10.2 Кэширование прав доступа

**`user_effective_permissions` — материализованная таблица**

**Преимущества:**
- Быстрая проверка прав (без join'ов)
- Используется в RLS политиках (`has_permission()`)

**Ограничения:**
- Требует ручного обновления при изменении permissions
- Обновляется через триггеры (может быть задержка)

**Как обновить вручную:**
```sql
SELECT refresh_user_effective_permissions('user-uuid');
```

### 10.3 Unified Assessment (объединённая оценка)

**Критическая логика в `UnifiedAssessmentPage.tsx`:**

**Загрузка вопросов:**
- Использует **`quality_id`** для `soft_skill_questions` (не `soft_skill_id`)
- Использует **`skill_id`** для `hard_skill_questions` (не `hard_skill_id`)

**Фильтрация:**
- Вопросы фильтруются на клиенте по `quality_id` или `skill_id`
- IDs берутся из `grade_qualities` и `grade_skills` для грейда пользователя

**Сохранение:**
- `is_draft=true` — автосохранение
- `is_draft=false` — финальная отправка (запускает агрегацию)

### 10.4 Триггеры и каскадное удаление

**При удалении пользователя:**
- Каскадно удаляются:
  - `user_roles`
  - `diagnostic_stage_participants`
  - `meeting_stage_participants`
  - `survey_360_assignments`
  - `soft_skill_results`, `hard_skill_results`
  - `user_assessment_results`
  - `tasks`
  - `one_on_one_meetings`
  - `user_career_progress`

**Логирование:**
- Edge function `delete-user` логирует удаление в `audit_log` перед удалением

### 10.5 RLS политики и SECURITY DEFINER

**Все helper функции используют `SECURITY DEFINER`:**
- `has_permission()` — выполняется с правами владельца
- `is_users_manager()` — минует RLS на `users`
- `refresh_user_effective_permissions()` — может обновлять любые записи

**Риск:**
- SQL-инъекции через параметры функций
- Обход RLS при неправильной реализации

**Защита:**
- `SET search_path = public` — защита от search_path атак
- Проверка входных параметров
- Использование `STABLE` вместо `VOLATILE` где возможно

### 10.6 Периоды оценки (evaluation_period)

**Формат:** `H1_YYYY` или `H2_YYYY`

**Автоматическое определение:**
- Триггер `set_evaluation_period` на `soft_skill_results` и `hard_skill_results`
- Логика:
  - Месяц 1-6 → H1
  - Месяц 7-12 → H2
- Используется для группировки результатов по полугодиям

### 10.7 Ограничения архитектуры

**1. Нет soft delete:**
- Удаление — физическое (через `DELETE`)
- Логи аудита хранят информацию об удалённых записях

**2. Нет версионирования данных:**
- История изменений в `audit_log` (только для пользователей)
- Для других сущностей — нет истории

**3. Нет транзакций на уровне приложения:**
- Триггеры выполняются в рамках транзакции БД
- Edge functions могут не успеть откатиться при ошибке

**4. Зависимость от внешнего API:**
- Yandex Cloud Functions для шифрования/дешифрования
- Если сервис недоступен — данные пользователей не читаются
