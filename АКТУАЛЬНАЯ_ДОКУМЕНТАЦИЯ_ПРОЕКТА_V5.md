## 2. БАЗА ДАННЫХ: ПОЛНАЯ СХЕМА

### 2.1 Аутентификация и пользователи

#### `auth.users` (управляется Supabase)
Встроенная таблица аутентификации Supabase.

#### `users`
Основная таблица пользователей (расширяет auth.users).

```sql
CREATE TABLE users (
  id UUID PRIMARY KEY,                    -- Совпадает с auth.users.id
  email TEXT NOT NULL,                    -- Зашифровано
  first_name TEXT,                        -- Зашифровано
  last_name TEXT,                         -- Зашифровано
  middle_name TEXT,                       -- Зашифровано
  employee_number TEXT UNIQUE NOT NULL,
  phone TEXT,
  department_id UUID REFERENCES departments(id),
  position_id UUID REFERENCES positions(id),
  grade_id UUID REFERENCES grades(id),
  manager_id UUID REFERENCES users(id),
  trade_point_id UUID REFERENCES trade_points(id),
  hire_date DATE,
  birth_date DATE,
  status BOOLEAN DEFAULT true,
  avatar_url TEXT,
  last_login_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);
```

#### `user_roles`
Связь пользователей и ролей.

```sql
CREATE TYPE app_role AS ENUM ('admin', 'hr_bp', 'manager', 'employee', 'moderator', 'user');

CREATE TABLE user_roles (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL,
  role app_role NOT NULL,
  created_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE(user_id, role)
);
```

**Роли:**
- `admin` - Полный доступ
- `hr_bp` - HR бизнес-партнёр
- `manager` - Руководитель
- `employee` - Сотрудник

### 2.5 Система вопросов и ответов

#### `answer_categories`
Категории ответов (определяют набор вариантов ответа).

```sql
CREATE TABLE answer_categories (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  description TEXT,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);
```

**Примеры категорий:**
- "Уровень владения навыком" (0-4)
- "Частота проявления качества" (Никогда, Редко, Иногда, Часто, Всегда)
- "Согласие с утверждением" (Совершенно не согласен ... Полностью согласен)

**Защита:** Функция `check_answer_category_usage` блокирует удаление категории, используемой в вопросах.

#### `hard_skill_answer_options`
Варианты ответов для вопросов по навыкам.

```sql
CREATE TABLE hard_skill_answer_options (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  title TEXT NOT NULL,
  label TEXT,                           -- Краткая подпись (например, "0", "1", "2")
  description TEXT,
  numeric_value INTEGER NOT NULL,       -- Числовое значение (для расчётов)
  level_value INTEGER DEFAULT 0,        -- Уровень компетенции (0-4)
  order_index INTEGER DEFAULT 1,        -- Порядок отображения
  answer_category_id UUID REFERENCES answer_categories(id),
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);
```

**Важно:** 
- `numeric_value` используется для вычисления среднего балла
- `level_value` соответствует уровню компетенции (0-4)
- Один вопрос может иметь несколько вариантов ответа из одной категории

### 2.7 Назначения и результаты оценки

#### `survey_360_assignments`
Назначения на оценку 360° (объединённая таблица для навыков и качеств).

```sql
CREATE TABLE survey_360_assignments (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  evaluated_user_id UUID NOT NULL REFERENCES users(id),
  evaluating_user_id UUID NOT NULL REFERENCES users(id),
  diagnostic_stage_id UUID REFERENCES diagnostic_stages(id),
  assignment_type TEXT,                 -- 'self', 'manager', 'peer'
  status TEXT DEFAULT 'pending',        -- 'pending', 'approved', 'rejected', 'completed'
  is_manager_participant BOOLEAN,
  assigned_date TIMESTAMPTZ DEFAULT now(),
  approved_by UUID REFERENCES users(id),
  approved_at TIMESTAMPTZ,
  rejected_at TIMESTAMPTZ,
  rejection_reason TEXT,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE(evaluated_user_id, evaluating_user_id)
);
```

**assignment_type:**
- `self` - Самооценка (evaluated = evaluating)
- `manager` - Оценка руководителем
- `peer` - Оценка коллегой

**Триггеры:**
- `auto_assign_manager_for_360` - автоматически создаёт назначение руководителя при создании самооценки
- `create_task_on_assignment_approval` - создаёт задачу при утверждении назначения
- `update_assignment_on_survey_completion` - обновляет статус при завершении опроса

#### `hard_skill_results`
Результаты оценки профессиональных навыков.

```sql
CREATE TABLE hard_skill_results (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  evaluated_user_id UUID NOT NULL REFERENCES users(id),
  evaluating_user_id UUID REFERENCES users(id),
  question_id UUID NOT NULL REFERENCES hard_skill_questions(id),
  answer_option_id UUID NOT NULL REFERENCES hard_skill_answer_options(id),
  comment TEXT,
  is_anonymous_comment BOOLEAN DEFAULT true,
  is_draft BOOLEAN DEFAULT true,
  assignment_id UUID REFERENCES survey_360_assignments(id),
  diagnostic_stage_id UUID REFERENCES diagnostic_stages(id),
  evaluation_period TEXT,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);
```

**Триггеры:**
- `set_evaluation_period` - автоматически устанавливает evaluation_period
- `update_user_skills_from_survey` - обновляет user_skills при сохранении (is_draft = false)
- `aggregate_hard_skill_results` - агрегирует результаты в user_assessment_results
- `complete_diagnostic_task_on_surveys_completion` - помечает задачу выполненной
- `update_assignment_on_survey_completion` - обновляет статус назначения

#### `soft_skill_results`
Результаты оценки 360° (личностные качества).

## 3. СИСТЕМА ВОПРОСОВ И ОТВЕТОВ

### 3.1 Архитектура системы

Система вопросов построена на концепции **категорий ответов** (`answer_categories`), которые определяют набор возможных вариантов ответа.

**Пример категории:** "Уровень владения навыком"

**Варианты ответов для этой категории:**
```
0 - Не владеет (level_value: 0, numeric_value: 0)
1 - Базовый (level_value: 1, numeric_value: 1)
2 - Средний (level_value: 2, numeric_value: 2)
3 - Продвинутый (level_value: 3, numeric_value: 3)
4 - Экспертный (level_value: 4, numeric_value: 4)
```

### 3.2 Поля вариантов ответа

#### `hard_skill_answer_options` и `soft_skill_answer_options`

- **title / label**: Текст варианта ответа, отображаемый пользователю
- **numeric_value**: Числовое значение для расчёта среднего балла
- **level_value**: Уровень компетенции (0-4), соответствует `competency_levels`
- **order_index**: Порядок отображения вариантов
- **answer_category_id**: FK на категорию ответов

### 3.3 Логика level_value (0-4)

**level_value** представляет уровень владения компетенцией:

| level_value | Название | Описание |
|-------------|----------|----------|
| 0 | Не владеет | Отсутствие навыка/качества |
| 1 | Базовый | Начальное владение |
| 2 | Средний | Уверенное владение |
| 3 | Продвинутый | Глубокое понимание |
| 4 | Экспертный | Максимальный уровень |

**Использование:**
- При сохранении результата (`hard_skill_results`, `soft_skill_results`) записывается `answer_option_id`
- Через JOIN с `answer_options` извлекается `level_value`
- `level_value` используется для:
  - Отображения на розе компетенций
  - Расчёта gap-анализа (разница между текущим и целевым уровнем)
  - Агрегации в `user_assessment_results`

### 3.4 Категории ответов по типам оценки

#### Для профессиональных навыков (hard skills):
- **Уровень владения**: 0 (Не владеет) ... 4 (Экспертный)
- **Частота применения**: Никогда, Редко, Иногда, Часто, Постоянно

#### Для личностных качеств (soft skills):
- **Частота проявления**: Никогда, Редко, Иногда, Часто, Всегда
- **Степень согласия**: Совершенно не согласен ... Полностью согласен
- **Оценка поведения**: Не проявляет ... Проявляет постоянно

### 3.5 Связь вопросов и компетенций

```
hard_skill_questions
  ├─ question_text (TEXT)
  ├─ skill_id (FK → hard_skills)
  ├─ answer_category_id (FK → answer_categories)
  └─ order_index

soft_skill_questions
  ├─ question_text (TEXT)
  ├─ quality_id (FK → soft_skills)
  ├─ category (TEXT) - для группировки в UI
  ├─ behavioral_indicators (TEXT)
  ├─ answer_category_id (FK → answer_categories)
  └─ order_index
```

**Один навык/качество может иметь несколько вопросов.**

**Примеры:**
- Навык "Python" → 3 вопроса с разными аспектами (синтаксис, OOP, асинхронность)
- Качество "Коммуникация" → 5 вопросов о различных ситуациях общения

### 4.1 Создание диагностического этапа

**Участники:** HR BP или Администратор

**Шаги:**

1. **Создание родительского этапа** (`parent_stages`)

   - Устанавливается is_active = true

   - Привязывается к parent_stage
   - Автоматически устанавливается evaluation_period через `get_evaluation_period()`
   - Статус: 'setup'
   - Прогресс: 0%

3. **Добавление участников** (`diagnostic_stage_participants`)

### 4.2 Прохождение оценки участником

**Шаг 1: Самооценка**

1. Пользователь видит задачу "Пройти самооценку" в своём списке задач
2. Переходит на страницу `/assessment/:assignmentId`
3. Загружается `UnifiedAssessmentPage`:

   - Выбирает вариант ответа из доступных
   - Может оставить комментарий (опционально)

5. При нажатии "Сохранить черновик":
   ```typescript
   await supabase.from('hard_skill_results').upsert({
     evaluated_user_id,
     evaluating_user_id,
   ```

   - `is_draft` устанавливается в `false`

   - Статус назначения меняется на 'completed'
   - Задача помечается как 'completed'

**Шаг 2: Выбор коллег**

После завершения самооценки:

1. Открывается `ColleagueSelectionDialog`
2. Пользователь выбирает 3-5 коллег для оценки
3. Создаются назначения:

4. Для каждого коллеги создаётся:

### 4.3 Агрегация результатов

После каждого сохранения результата (`is_draft = false`) запускаются триггеры:

**Триггер `aggregate_hard_skill_results`:**
```sql
-- Удаляет старые агрегированные данные
DELETE FROM user_assessment_results
WHERE user_id = NEW.evaluated_user_id
  AND diagnostic_stage_id = stage_id
  AND skill_id IS NOT NULL;

-- Создаёт новые агрегированные данные
INSERT INTO user_assessment_results (
  user_id, diagnostic_stage_id, skill_id,
  self_assessment, manager_assessment, peers_average, total_responses
)
SELECT 
  NEW.evaluated_user_id,
  stage_id,
  hq.skill_id,
  AVG(CASE WHEN evaluating = evaluated THEN numeric_value END) as self,
  AVG(CASE WHEN evaluating = manager THEN numeric_value END) as manager,
  AVG(CASE WHEN evaluating NOT IN (evaluated, manager) THEN numeric_value END) as peers,
  COUNT(*)
FROM hard_skill_results sr
JOIN hard_skill_questions hq ON sr.question_id = hq.id
JOIN hard_skill_answer_options ao ON sr.answer_option_id = ao.id
WHERE sr.evaluated_user_id = NEW.evaluated_user_id
  AND sr.diagnostic_stage_id = stage_id
  AND sr.is_draft = false
GROUP BY hq.skill_id;
```

**Аналогично для `aggregate_soft_skill_results`.**

### 4.4 Обновление прогресса этапа

**Функция `calculate_diagnostic_stage_progress`:**
```sql
SELECT COUNT(*) FROM diagnostic_stage_participants WHERE stage_id = stage_id_param;
-- total_participants

-- Считаем завершённые оценки навыков
SELECT COUNT(DISTINCT evaluated_user_id) FROM hard_skill_results
WHERE diagnostic_stage_id = stage_id_param AND is_draft = false;

-- Считаем завершённые оценки качеств
SELECT COUNT(DISTINCT evaluated_user_id) FROM soft_skill_results
WHERE diagnostic_stage_id = stage_id_param AND is_draft = false;

-- Прогресс = ((completed_skills + completed_qualities) / (total_participants * 2)) * 100
```

**Триггер `update_diagnostic_stage_on_participant_add`:**
- Вызывается при добавлении/изменении результатов
- Пересчитывает прогресс через `calculate_diagnostic_stage_progress`
- Обновляет статус этапа:
  - 0% → 'setup'
  - 1-99% → 'assessment'
  - 100% → 'completed'

## 5. ЛОГИКА ОЦЕНКИ: SELF/MANAGER/PEER

### 5.1 Типы оценивающих

В системе существует 3 типа оценивающих, определяемых полем `assignment_type` в `survey_360_assignments`:

| assignment_type | Описание | Условие |
|----------------|----------|---------|
| `self` | Самооценка | `evaluated_user_id = evaluating_user_id` |
| `manager` | Оценка руководителем | `assignment_type = 'manager'` OR `is_manager_participant = true` |
| `peer` | Оценка коллегой | `assignment_type = 'peer'` AND `evaluated_user_id != evaluating_user_id` |

### 5.2 Определение типа оценивающего

**В коде (TypeScript):**

```typescript
// Загрузка назначения
const { data: assignment } = await supabase
  .from('survey_360_assignments')
  .select('*')
  .eq('id', assignmentId)
  .single();

// Определение типа
const isSelf = assignment.evaluated_user_id === currentUser.id;
const isManager = assignment.assignment_type === 'manager' || 
                  assignment.is_manager_participant === true;

const evaluatorType = isSelf ? 'self' : isManager ? 'manager' : 'peer';
```

**В SQL (для агрегации):**

```sql
-- Самооценка
AVG(CASE WHEN sr.evaluating_user_id = sr.evaluated_user_id THEN ao.numeric_value END) as self_assessment

-- Оценка руководителя
AVG(CASE WHEN sr.evaluating_user_id = manager_id THEN ao.numeric_value END) as manager_assessment

-- Оценка коллег (среднее)
AVG(CASE 
  WHEN sr.evaluating_user_id != sr.evaluated_user_id 
    AND (manager_id IS NULL OR sr.evaluating_user_id != manager_id)
  THEN ao.numeric_value 
END) as peers_average
```

### 5.3 Права доступа к результатам

**RLS политика `hard_skill_results_select_auth_policy`:**
```sql
CREATE POLICY hard_skill_results_select_auth_policy ON hard_skill_results
FOR SELECT
USING (
  evaluated_user_id = auth.uid() OR              -- Может видеть свои результаты
  evaluating_user_id = auth.uid() OR             -- Может видеть свои оценки других
  EXISTS (
    SELECT 1 FROM users 
    WHERE id = hard_skill_results.evaluated_user_id 
      AND manager_id = auth.uid()                -- Руководитель видит результаты подчинённых
  ) OR
  EXISTS (
    SELECT 1 FROM user_roles 
    WHERE user_id = auth.uid() 
      AND role IN ('admin', 'hr_bp')             -- Администраторы и HR видят всё
  )
);
```

### 5.4 Анонимность комментариев

**Поле `is_anonymous_comment` в `hard_skill_results` и `soft_skill_results`:**

- **По умолчанию:** `true` (анонимно)
- **Для самооценки (self):** `false` (не анонимно)
- **Для руководителя (manager):** `false` (не анонимно)
- **Для коллег (peer):** `true` (анонимно)

**Логика отображения комментария:**

```typescript

  authorName = result.evaluating_user_name;
}

// Исключение: пользователь видит свои комментарии неанонимно
if (result.evaluating_user_id === currentUser.id) {
  authorName = "Вы";
}
```

```typescript
// src/hooks/useSkills.ts
import { useQuery, useMutation, useQueryClient } from '@tanstack/react-query';
import { supabase } from '@/integrations/supabase/client';

export const useSkills = () => {
  const queryClient = useQueryClient();

  // Загрузка данных
  const { data: skills, isLoading } = useQuery({
    queryKey: ['hard_skills'],
    queryFn: async () => {
      const { data, error } = await supabase
        .from('hard_skills')
        .select('*')
        .order('name', { ascending: true });

      if (error) throw error;
      return data;
    },
  });

  // Создание
  const createSkillMutation = useMutation({
    mutationFn: async (skill) => {
      const { data, error } = await supabase
        .from('hard_skills')
        .insert(skill)
        .select()
        .single();

      if (error) throw error;
      return data;
    },
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ['hard_skills'] });
      toast({ title: 'Навык создан' });
    },
  });

  return {
    skills,
    isLoading,
    createSkill: createSkillMutation.mutate,
  };
};
```

### 7.7 Edge Functions

**Путь:** `supabase/functions/`

#### `create-user`
Создание пользователя с ролью.

**Вход:**
```json
{
  "email": "user@example.com",
  "password": "password",
  "userData": {
    "first_name": "Иван",
    "last_name": "Иванов",
    "employee_number": "EMP001",

  }

```

**Логика:**
1. Создаёт пользователя в `auth.users` через `supabase.auth.admin.createUser()`
2. Вставляет запись в `users` с тем же `id`
3. Создаёт запись в `user_roles`
4. Логирует действие в `admin_activity_logs`

#### `delete-user`
Удаление пользователя.

**Логика:**
1. Удаляет пользователя из `auth.users`
2. Каскадно удаляются связанные записи (через FK constraints)

#### `create-admin`
Создание администратора.

**Логика:**
Аналогична `create-user`, но с ролью `admin`.

#### `generate-development-tasks`
Генерация задач развития на основе gap-анализа.

**Вход:**
```json
{
  "userId": "...",
  "gradeId": "..."
}
```

**Логика:**
1. Вызывает `calculate_career_gap(userId, gradeId)`
2. Для каждого навыка/качества с gap > 0:
   - Находит подходящие `development_tasks`
   - Создаёт задачу для пользователя

#### `create-peer-evaluation-tasks`
Создание задач для peer-оценки.

**Логика:**
1. Получает список `survey_360_assignments` со статусом 'approved'
2. Для каждого назначения создаёт задачу в `tasks`

### 8.1 Row Level Security (RLS)

Все таблицы защищены политиками RLS.

#### Примеры политик

**users:**
```sql
-- Пользователь видит себя, своих подчинённых и коллег
CREATE POLICY users_select_policy ON users
FOR SELECT
USING (
  id = auth.uid() OR
  manager_id = auth.uid() OR
  department_id IN (
    SELECT department_id FROM users WHERE id = auth.uid()
  ) OR
  has_permission(auth.uid(), 'users.view')
);

-- Обновлять может только администратор или сам пользователь (ограниченно)
CREATE POLICY users_update_policy ON users
FOR UPDATE
USING (
  id = auth.uid() OR
  has_permission(auth.uid(), 'security.manage_users')
);
```

**hard_skill_results:**
```sql
-- Вставлять может только сам оценивающий
CREATE POLICY hard_skill_results_insert_auth_policy ON hard_skill_results

WITH CHECK (evaluating_user_id = auth.uid());

-- Обновлять может только сам оценивающий или admin/HR
CREATE POLICY hard_skill_results_update_auth_policy ON hard_skill_results
FOR UPDATE
USING (
```

```sql
CREATE FUNCTION log_access_denied(
  _permission_name TEXT,
  _resource_type TEXT DEFAULT NULL,
  _resource_id UUID DEFAULT NULL,
  _action_attempted TEXT DEFAULT NULL
)
RETURNS VOID
SECURITY DEFINER
AS $$
DECLARE
  current_user_id UUID;

      current_user_id, _permission_name, _resource_type, _resource_id,

    );
  END IF;
END;
$$ LANGUAGE plpgsql;
```

#### `refresh_user_effective_permissions(target_user_id)`

Обновляет кэш разрешений пользователя.

```sql
CREATE FUNCTION refresh_user_effective_permissions(target_user_id UUID)
RETURNS VOID
SECURITY DEFINER
AS $$
BEGIN
  -- Удаляем старые разрешения
  DELETE FROM user_effective_permissions WHERE user_id = target_user_id;
  
  -- Вставляем новые разрешения на основе ролей
  INSERT INTO user_effective_permissions (user_id, permission_name)
  SELECT DISTINCT
    target_user_id,
    p.name
  FROM user_roles ur
  JOIN role_permissions rp ON rp.role = ur.role
  JOIN permissions p ON p.id = rp.permission_id
  WHERE ur.user_id = target_user_id
  ON CONFLICT (user_id, permission_name) DO NOTHING;
END;
$$ LANGUAGE plpgsql;
```

#### `refresh_role_effective_permissions(target_role)`

Обновляет разрешения для всех пользователей с данной ролью.

```sql
CREATE FUNCTION refresh_role_effective_permissions(target_role app_role)
RETURNS VOID
SECURITY DEFINER
AS $$
DECLARE
  user_record RECORD;
BEGIN
  FOR user_record IN 
    SELECT DISTINCT user_id FROM user_roles WHERE role = target_role
  LOOP
    PERFORM refresh_user_effective_permissions(user_record.user_id);
  END LOOP;
END;
$$ LANGUAGE plpgsql;
```

### 9.3 Функции диагностики

#### `calculate_diagnostic_stage_progress(stage_id_param)`

Вычисляет прогресс этапа диагностики.

```sql
CREATE FUNCTION calculate_diagnostic_stage_progress(stage_id_param UUID)
RETURNS NUMERIC
SECURITY DEFINER
AS $$
DECLARE
  total_participants INTEGER;
  completed_skill_surveys INTEGER;
  completed_360_surveys INTEGER;
  total_required INTEGER;
  completed_total INTEGER;
  progress NUMERIC;
BEGIN
  -- Всего участников
  SELECT COUNT(*) INTO total_participants
  FROM diagnostic_stage_participants
  WHERE stage_id = stage_id_param;
  
  IF total_participants = 0 THEN
    RETURN 0;
  END IF;
  
  -- Требуется выполнить: 2 оценки на участника (навыки + качества)
  total_required := total_participants * 2;
  
  -- Завершённые оценки навыков
  SELECT COUNT(DISTINCT evaluated_user_id) INTO completed_skill_surveys
  FROM hard_skill_results
  WHERE diagnostic_stage_id = stage_id_param
    AND is_draft = false;
  
  -- Завершённые оценки качеств
  SELECT COUNT(DISTINCT evaluated_user_id) INTO completed_360_surveys
  FROM soft_skill_results
  WHERE diagnostic_stage_id = stage_id_param
    AND is_draft = false;
  
  completed_total := completed_skill_surveys + completed_360_surveys;
  progress := (completed_total::NUMERIC / total_required::NUMERIC) * 100;
  
  RETURN ROUND(progress, 2);
END;
$$ LANGUAGE plpgsql;
```

```sql
CREATE FUNCTION create_diagnostic_task_for_participant()

DECLARE
  stage_record RECORD;

  participant_full_name TEXT;
  self_assignment_id UUID;
  manager_assignment_id UUID;
BEGIN
  -- Получаем информацию об этапе

  SELECT id INTO self_assignment_id
  FROM survey_360_assignments

    AND evaluating_user_id = NEW.user_id
    AND diagnostic_stage_id = NEW.stage_id
    AND assignment_type = 'self';
  
  -- Создаём задачу для участника
  IF self_assignment_id IS NOT NULL THEN

      NEW.user_id, NEW.stage_id, self_assignment_id, 'self',

  -- Создаём задачу для руководителя
  IF manager_user_id IS NOT NULL THEN
    SELECT id INTO manager_assignment_id
    FROM survey_360_assignments
    WHERE evaluated_user_id = NEW.user_id
      AND evaluating_user_id = manager_user_id
      AND diagnostic_stage_id = NEW.stage_id

    SELECT CONCAT(last_name, ' ', first_name, ' ', COALESCE(middle_name, ''))
```

### 9.5 Триггеры агрегации результатов

#### `aggregate_hard_skill_results`

Агрегирует результаты навыков.

```sql
CREATE TRIGGER aggregate_hard_skill_results_trigger
AFTER INSERT OR UPDATE ON hard_skill_results
FOR EACH ROW

CREATE FUNCTION aggregate_hard_skill_results()
RETURNS TRIGGER AS $$
DECLARE
  stage_id UUID;
  manager_id UUID;
BEGIN
  stage_id := NEW.diagnostic_stage_id;
  IF stage_id IS NULL THEN RETURN NEW; END IF;
  
  SELECT u.manager_id INTO manager_id FROM users u WHERE u.id = NEW.evaluated_user_id;
  
  -- Удаляем старые данные
  DELETE FROM user_assessment_results
  WHERE user_id = NEW.evaluated_user_id
    AND diagnostic_stage_id = stage_id
    AND skill_id IS NOT NULL;
  
  -- Вставляем новые агрегированные данные
  INSERT INTO user_assessment_results (
    user_id, diagnostic_stage_id, assessment_period, assessment_date, skill_id,
    self_assessment, manager_assessment, peers_average, total_responses
  )
  SELECT 
    NEW.evaluated_user_id, stage_id, get_evaluation_period(NOW()), NOW(), hq.skill_id,
    AVG(CASE WHEN sr.evaluating_user_id = NEW.evaluated_user_id THEN ao.numeric_value END),
    AVG(CASE WHEN sr.evaluating_user_id = manager_id THEN ao.numeric_value END),
    AVG(CASE 
      WHEN sr.evaluating_user_id != NEW.evaluated_user_id 
        AND (manager_id IS NULL OR sr.evaluating_user_id != manager_id)
      THEN ao.numeric_value
    END),
    COUNT(*)
  FROM hard_skill_results sr
  JOIN hard_skill_questions hq ON sr.question_id = hq.id
  JOIN hard_skill_answer_options ao ON sr.answer_option_id = ao.id
  WHERE sr.evaluated_user_id = NEW.evaluated_user_id
    AND sr.diagnostic_stage_id = stage_id
    AND sr.is_draft = false
    AND hq.skill_id IS NOT NULL
  GROUP BY hq.skill_id;
  
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

### 9.7 Функции карьерного развития

#### `calculate_career_gap(p_user_id, p_grade_id)`

Вычисляет gap между текущими и целевыми компетенциями.

```sql
CREATE FUNCTION calculate_career_gap(p_user_id UUID, p_grade_id UUID)
RETURNS TABLE(
  competency_type TEXT,
  competency_id UUID,
  competency_name TEXT,
  current_level NUMERIC,
  target_level NUMERIC,
  gap NUMERIC,
  is_ready BOOLEAN
)
SECURITY DEFINER
AS $$
BEGIN
  -- Навыки
  RETURN QUERY
  SELECT 
    'skill'::TEXT,
    gs.skill_id,
    s.name,
    COALESCE(uar.self_assessment, 0),
    gs.target_level,
    (gs.target_level - COALESCE(uar.self_assessment, 0)),
    (COALESCE(uar.self_assessment, 0) >= gs.target_level)
  FROM grade_skills gs
  JOIN hard_skills s ON s.id = gs.skill_id
  LEFT JOIN (
    SELECT skill_id, AVG(self_assessment) as self_assessment
    FROM user_assessment_results
    WHERE user_id = p_user_id AND skill_id IS NOT NULL
    GROUP BY skill_id
  ) uar ON uar.skill_id = gs.skill_id
  WHERE gs.grade_id = p_grade_id;

  -- Качества
  RETURN QUERY
  SELECT 
    'quality'::TEXT,
    gq.quality_id,
    q.name,
    COALESCE(uar.self_assessment, 0),
    gq.target_level,
    (gq.target_level - COALESCE(uar.self_assessment, 0)),
    (COALESCE(uar.self_assessment, 0) >= gq.target_level)
  FROM grade_qualities gq
  JOIN soft_skills q ON q.id = gq.quality_id
  LEFT JOIN (
    SELECT quality_id, AVG(self_assessment) as self_assessment
    FROM user_assessment_results
    WHERE user_id = p_user_id AND quality_id IS NOT NULL
    GROUP BY quality_id
  ) uar ON uar.quality_id = gq.quality_id
  WHERE gq.grade_id = p_grade_id;
END;
$$ LANGUAGE plpgsql;
```

#### `recommend_career_tracks(p_user_id, p_limit)`

Рекомендует карьерные треки.

```sql
CREATE FUNCTION recommend_career_tracks(p_user_id UUID, p_limit INTEGER DEFAULT 5)
RETURNS TABLE(
  track_id UUID,
  track_name TEXT,
  track_type_name TEXT,
  target_position_name TEXT,
  compatibility_score NUMERIC,
  total_gap NUMERIC,
  steps_count INTEGER
)
SECURITY DEFINER
AS $$
DECLARE
  v_user_position_id UUID;
  v_user_grade_id UUID;
BEGIN
  -- Получаем текущую позицию и грейд
  SELECT position_id, grade_id 
  INTO v_user_position_id, v_user_grade_id
  FROM users 
  WHERE id = p_user_id;

  -- Подбираем треки
  RETURN QUERY
  WITH track_stats AS (
    SELECT 
      ct.id, ct.name, tt.name as type_name, p.name as position_name,
      COUNT(cts.id) as steps_count,
      (CASE WHEN ct.target_position_id = v_user_position_id THEN 50 ELSE 20 END +
       CASE WHEN COUNT(cts.id) > 0 THEN 30 ELSE 0 END)::NUMERIC as score
    FROM career_tracks ct
    LEFT JOIN track_types tt ON tt.id = ct.track_type_id
    LEFT JOIN positions p ON p.id = ct.target_position_id
    LEFT JOIN career_track_steps cts ON cts.career_track_id = ct.id
    GROUP BY ct.id, ct.name, tt.name, p.name, ct.target_position_id
    HAVING COUNT(cts.id) > 0
  ),
  track_gaps AS (
    SELECT 
      ts.*, COALESCE(SUM(ABS(gap_data.gap)), 0) as total_gap
    FROM track_stats ts
    LEFT JOIN career_track_steps cts ON cts.career_track_id = ts.id
    LEFT JOIN LATERAL (
      SELECT SUM(ABS(gap)) as gap
      FROM calculate_career_gap(p_user_id, cts.grade_id)
    ) gap_data ON true
    GROUP BY ts.id, ts.name, ts.type_name, ts.position_name, ts.score, ts.steps_count
  )
  SELECT 
    tg.id, tg.name, tg.type_name, tg.position_name,
    tg.score, tg.total_gap, tg.steps_count
  FROM track_gaps tg
  ORDER BY tg.score DESC, tg.total_gap ASC
  LIMIT p_limit;
END;
$$ LANGUAGE plpgsql;
```
